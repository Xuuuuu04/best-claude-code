---
description: 针对已知问题（bug / 审查打回 / 测试失败 / 安全 finding）执行可追溯的最小化修复流水线（开发组长定位 → 开发 Agent 修复 → 代码审计师 → 复测）。禁止借机重构。
argument-hint: [必填：问题描述或报告文件路径，例如 reviews/review-task001-v1.md]
---

<command>

<section id="rules">
<title>铁律</title>
<content>
1. **`$ARGUMENTS` is required**: This command MUST have a specific issue description or report file path. Empty invocation = refuse and ask user to provide the problem.
2. **Minimal fix scope only**: Fix the specified problem. Do NOT refactor adjacent code, improve naming, or add features while fixing. Scope creep in fixes creates untraceable regressions.
3. **No new dependencies**: Fixes MUST NOT introduce new dependencies unless the dependency IS the fix (e.g., replacing a vulnerable library).
4. **Interface signature preservation**: Do not change function/API signatures unless the bug IS in the interface. Breaking changes require /新功能 pipeline.
5. **Cross-module / interface-breaking = escalate**: If the fix requires changes across multiple modules or breaks an interface contract, STOP and escalate to /新功能.
</content>
</section>

<section id="scope">
<title>本 command 覆盖范围</title>
<content>
**适用场景**
- Bug 修复（描述或 Issue URL）
- 代码审计师打回项（reviews/review-taskXXX-vN.md）
- 功能测试失败项（test-reports/test-taskXXX.md）
- 安全审计 finding（security-audits/audit-*.md）
- 界面测试截图不符（ui-reviews/ui-*.md）

**不适用**（用其他命令）
- 新增功能 → /新功能
- 跨模块/破坏接口的修复 → /新功能
- 纯重命名/格式化 → 直接派开发 Agent，不走本命令
</content>
</section>

<section id="dispatch-flow">
<title>调度流水线</title>
<content>
**Step 1 — 问题锚定**（主进程执行）
- 前置：`$ARGUMENTS` 不为空，否则 STOP
- 读源文件，提取：严重度 / 涉及文件+行号 / 根因假设 / 建议修复方向
- 产出结构化问题摘要

**Step 2 — 根因定位**（派 开发组长 Agent）
- 目的：弱模型容易把"症状"当"根因"，dev-lead 裁决修复落在哪个文件
- 产出：根因定位（一句话）/ 修复点清单（文件:行号）/ 推荐修复 Agent / 影响半径（测试/文档/schema）

**Step 3 — 执行修复**（派 开发组长 推荐的开发 Agent）
- 候选：backend / frontend / database / miniprogram-dev
- 硬性约束：只改必要代码 / 不借机重构 / 不新增依赖 / 不改接口签名（除非 bug 在接口）
- 输入：问题原文 + dev-lead 定位结论

**Step 4 — 代码审计师复核**（必须，不可跳过）
- Checklist：修复真的解决问题（非掩盖症状）/ 未引入新问题 / 未违反 GP-C* / 修复范围未过度

**Step 5 — 按来源复测**（必须，不可跳过）
- 功能测试失败 → 功能测试师重跑原用例
- 安全审计 finding → 安全审计师复核
- 界面测试问题 → 界面测试师重新截图
- 代码审计打回 → 代码审计师终验（合并入 Step 4）

**Step 6 — 汇报**（主进程执行）
- 根因 / 修复内容 / 影响文件
- 审查结论 + 复测结论
- 后续建议（rollback / 补文档 / 通知 client）
</content>
</section>

<section id="final_reminder">
<title>Final Reminder</title>
<content>
- No `$ARGUMENTS` = refuse to start. Problem must be specified.
- Minimal fix scope — no refactoring, no new features, no dependency addition.
- 代码审计师 (Step 4) and corresponding retest (Step 5) are mandatory — no skips.
- Cross-module or interface-breaking fix → stop, escalate to /新功能.
- Fix the symptom's root cause, not just the surface symptom.
</content>
