argument-hint: [可选：目标目录路径，默认当前工作目录]

<section id="rules">
<title>铁律</title>
<content>
1. **Never write to `~/.claude/` core files**: Hook-A physically blocks writes to `~/.claude/` (except whitelisted paths). This command operates on project directories only.
2. **Empty argument + home directory = STOP**: If `$ARGUMENTS` is empty AND the current directory is `~` or `~/.claude`, refuse and ask user to specify a project directory or `cd` into one first.
3. **Project marker required**: Target directory must contain at least one of `.git` / `package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod`. Warn and ask before proceeding without one.
4. **Read before write**: Existing CLAUDE.md files MUST be read before updating. Content inside `<!-- user-content -->` markers must be preserved unchanged.
5. **Single CLAUDE.md ≤ 200 lines**: If content would exceed 200 lines, trim the file index section. Quality over quantity.
6. **Exclude build artifacts**: Never include `node_modules` / `.git` / `dist` / `build` / `__pycache__` / `.next` / `target` / `vendor` / `.venv` / `coverage` / `.turbo` in analysis or file index.
7. **Preserve existing structure**: If a directory already has a well-maintained CLAUDE.md, prefer incremental update over full rewrite.
</content>
</section>

<section id="role">
<title>角色定位</title>
<content>
本 command 是项目知识图谱的构建者。它通过深度扫描项目目录树，为每一层代码目录生成/更新 CLAUDE.md，形成一套自描述的文档体系——让任何打开项目的人都能快速理解"这个目录是做什么的、用什么技术、关键文件有哪些"。

核心价值：将隐性的项目知识（目录结构、技术栈、架构约定）显性化、结构化、可检索，降低新成员的上手成本和长期维护的认知负担。

Unlike `/进度同步`：本 command 全面重建/创建 CLAUDE.md，而 `/进度同步` 只更新已有 CLAUDE.md 的"最近进度"段。Unlike `/目录整理`：本 command 分析并记录目录结构，但不修改文件位置。Unlike `/项目状态`：本 command 写入知识文档，而 `/项目状态` 读取运行时状态。
</content>
</section>

<section id="scope">
<title>职责边界（In-Scope）</title>
<content>
**本 command 负责的范围**：
1. 扫描项目目录树，识别所有含代码的目录层级
2. 为每层目录生成/更新 CLAUDE.md，包含：目录职责 / 技术栈 / 架构 / 对外接口 / 内部逻辑 / 依赖关系 / 文件索引
3. 在项目根目录额外产出：项目总览 / ASCII 目录树 / 全局技术栈汇总 / 模块依赖图
4. 识别并记录项目的技术栈、框架、构建工具
5. 提取并记录对外接口（API 端点 / 公共函数 / CLI 命令）
6. 生成文件索引（关键文件列表 + 一句话职责说明）
</content>
</section>

<section id="not-scope">
<title>不承担的职责（Out-of-Scope）</title>
<content>
1. **业务代码修改**：只创建/更新 CLAUDE.md，不修改任何业务代码文件
2. **运行时状态记录**：当前 Task 进度、阻塞项属于 `/进度同步` 或 `@pm` 的 progress-log
3. **目录结构整理**：移动/重命名/删除文件属于 `/目录整理`
4. **代码质量审查**：代码逻辑正确性、安全问题属于 `@code-review` / `@security-auditor`
5. **架构决策制定**：本 command 记录现有架构，不做架构改进建议 → 需 `@architect`
6. **`~/.claude/` 核心目录**：Hook-A 物理拦截，无法操作
</content>
</section>

<section id="methodology">
<title>方法论与执行步骤</title>
<content>
**Workflow — 6 步流水线**

**Step 1 — 边界校验**（主进程执行）
- 确定 target = `$ARGUMENTS` 或当前目录
- 校验三项边界：
  - B1：target 不是 `~/.claude/` 核心目录
  - B2：`$ARGUMENTS` 为空且当前目录是 `~` → STOP，请用户指定项目目录
  - B3：target 包含项目标记（`.git` / `package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod`）
- 未通过 → STOP，向用户说明原因

**Step 2 — 目录树扫描**（主进程执行）
- 用 Glob 扫描 target 下所有含代码目录
- 排除清单：`node_modules` / `.git` / `dist` / `build` / `__pycache__` / `.next` / `target` / `vendor` / `.venv` / `venv` / `coverage` / `.turbo` / `.claude` / `.config` / `.vscode` / `.idea`
- 产出：目录清单 + 每个目录是否已有 CLAUDE.md 的标记 + 技术栈初步识别

**Step 3 — 深度探索**（派 Explore Agent，最多 5 个一批）
- 每个 Agent 任务：
  1. 列出目录下所有文件（排除构建产物）
  2. 识别语言和框架（从文件扩展名和配置文件推断）
  3. 分析入口文件（`main.py` / `index.ts` / `App.vue` 等）
  4. 提取对外接口（API 路由 / 公共函数 / 导出的模块）
  5. 理解内部逻辑（核心类/函数的职责）
  6. 识别调用关系（import 依赖、模块间引用）
- 注意：Explore Agent 为只读，GP-O12 豁免授权（纯只读，无写依赖）

**Step 4 — 逐层写入 CLAUDE.md**（主进程执行）
- 模板内容（每层 CLAUDE.md 包含以下 section）：
  ```markdown
  ---
  name: {目录名}
  description: {一句话职责}
  ---

  ## 目录职责
  {该目录的核心职责和业务范围}

  ## 技术栈
  {语言 / 框架 / 构建工具 / 测试工具}

  ## 架构
  {架构模式 / 分层结构 / 核心设计}

  ## 对外接口
  {API 端点 / 公共函数 / CLI 命令}

  ## 内部逻辑概述
  {核心流程 / 关键算法 / 状态管理}

  ## 依赖关系
  {依赖的其他模块 / 被谁依赖}

  ## 文件索引
  | 文件 | 职责 |
  |------|------|
  | {文件名} | {一句话说明} |

  ## 最近进度
  <!-- 由 /进度同步 维护 -->
  ```
- 已有 CLAUDE.md 先读再改，保留 `<!-- user-content -->` 标记内容
- 已完善的只做增量补充（技术栈变更 / 新增接口 / 文件索引更新）

**Step 5 — 项目根统领**（主进程执行，仅项目根目录）
- 额外产出：
  - 项目总览（一句话描述项目是什么）
  - ASCII 目录树（主要模块层级）
  - 全局技术栈汇总（所有模块的技术栈合集）
  - 模块依赖图（文字描述各模块间的依赖关系）

**Step 6 — 汇报**
- 输出：扫描 N 个目录 / 新增 X / 更新 Y / 跳过 Z

**关键决策点**：
- 某目录无代码文件（只有文档或配置）→ 可选生成简短 CLAUDE.md 或跳过
- 内容超过 200 行 → 精简文件索引部分，保留核心 section
- 发现架构模式不明确 → 记录为"待确认"，不猜测

**常见反模式与对策**：
| 反模式 | 表现 | 对策 |
|--------|------|------|
| 过度生成 | 为每个子目录都生成 CLAUDE.md | 只生成含代码的目录，空目录跳过 |
| 信息过时 | 已有 CLAUDE.md 但内容已过时 | 增量更新，保留用户自定义内容 |
| 过度推断 | 猜测架构模式但不确定 | 记录为"待确认"，不虚构 |
| 忽视用户内容 | 覆盖了用户自定义的 CLAUDE.md 内容 | 铁律 4：保留 `<!-- user-content -->` 标记内内容 |
</content>
</section>

<section id="collaboration">
<title>协作边界</title>
<content>
**上游（谁触发本 command）**：
- 用户直接输入 `/项目分析 [可选路径]`
- 新接手项目时的初始化动作
- 架构发生重大变化后需要全量重建知识图谱

**下游（本 command 派谁）**：
- Step 3 派 Explore Agent（只读，批量并行，GP-O12 豁免）
- 不派其他专职 Agent

**横向衔接**：
- 与 `/进度同步`：本 command 创建 CLAUDE.md，`/进度同步` 更新其"最近进度"段
- 与 `/项目状态`：本 command 生成的 CLAUDE.md 是 `/项目状态` 读取的信息源之一
- 与 `/目录整理`：目录整理后应重新运行 `/项目分析` 更新文件索引
- 与 `@architect`：若发现架构模式与现有文档矛盾 → 建议用户咨询 `@architect`
</content>
