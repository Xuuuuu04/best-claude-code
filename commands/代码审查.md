<section id="rules">
<title>铁律</title>
<content>
1. **Serial execution enforced by Hook-E**: Each gate executes one at a time. Gate N+1 does not start until Gate N completes. Parallel gate execution is physically blocked.
2. **code-review + test-func + test-lead are non-skippable**: These three gates are mandatory for every invocation, regardless of change size or scope.
3. **security-auditor and test-ui skip requires ★ Insight documentation**: Conditional gates that are skipped MUST have the skip reason recorded in a ★ Insight block. Silent skip = protocol violation.
4. **Verdict "打回" terminates the command**: When test-lead issues a 打回 verdict, this command ends. Fixes go through /快速修复, not a retry of this command.
5. **Scope must be deterministic**: If `$ARGUMENTS` is not provided, the scope MUST be derived from `git diff --name-only` + `git diff --cached --name-only`. If no diff exists, stop and ask user.
</content>
</section>

<section id="scope">
<title>本 command 覆盖范围</title>
<content>
**审查范围确定**：`$ARGUMENTS` 给出文件/模块/Task ID → 按此范围；否则取 git diff 改动文件集

**4 层门控**：
- Gate 1：代码审计师（必须）
- Gate 2：安全审计师（条件触发）
- Gate 3：功能测试师（必须）
- Gate 4：界面测试师（条件触发，有前端改动时）
- 最终：测试总监师裁决（必须）
</content>
</section>

<section id="dispatch-flow">
<title>调度流水线</title>
<content>
**Step 1 — 确定审查范围**（主进程执行）
- 解析 `$ARGUMENTS` 或取 git diff
- 无改动 → STOP，要求用户确认或运行 git status

**Step 2 — 读取上下文**（主进程执行）
- 读相关 Task 文件（业务描述 + 技术方案）
- 读改动文件内容
- 读项目 CLAUDE.md 架构约定

**Step 3 — 代码审计师**（必须）
- 范围：per-diff 代码质量基线 + 安全基线 4 项（SQL 注入/XSS/硬编码凭据/命令注入）
- 输入：Task 文件 + 改动清单 + 改动内容
- 产出：分级问题清单（Critical / High / Medium / Low）
- 通过条件：无 Critical；High 附修复建议

**Step 4 — 安全审计师**（条件触发）
- 触发条件（任一）：改动涉及认证/授权/加密/会话/支付/文件上传 / 引入新依赖 / 代码审计师标记 security-escalate / 临近上线
- 范围：OWASP Top 10 + CWE Top 25 + 依赖 CVE + 密钥泄露 + 合规

**Step 5 — 功能测试师**（必须）
- 输入：Task 文件 + 审查报告
- 执行：端到端用户流程 + 边界值 + 异常路径
- 产出：结构化测试报告（通过/失败用例）

**Step 6 — 界面测试师**（条件触发，触发后不可跳过）
- 触发条件：改动涉及 *.tsx / *.vue / 前端路由 / CSS / 视觉组件
- 产出：截图 + interaction-check.md

**Step 7 — 测试总监师裁决**（必须，最终门控）
- 输入：上述全部报告 + UI 证据（若有）
- 通过：功能全绿 + UI 正常 + 安全无高危
- 有条件通过：功能全绿 + 安全无高危；UI 有瑕疵（记录条件与补齐时间）
- 打回：功能未全绿 或 发现高危（附可执行修复建议）

**Step 8 — 输出摘要**（主进程执行）
- 裁决结论 / 各层报告路径 / 问题数量 / 下一步建议
</content>
</section>
