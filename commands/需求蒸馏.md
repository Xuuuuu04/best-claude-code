---
description: 把一句话原始需求蒸馏为团队可执行的 Task 列表（客户沟通师语义增强 → 歧义澄清 → 项目管理师拆 Task）。大多数新需求的标准入口。
argument-hint: [必填：原始需求描述，一句话或几句话均可]
---

<command>

<section id="rules">
<title>铁律</title>
<content>
1. **`$ARGUMENTS` is required**: This command needs a raw requirement input. Empty invocation = refuse and prompt user to provide at least one sentence describing what they want.
2. **Ambiguity = stop, not guess**: When client-agent identifies ambiguity, the flow MUST pause and ask the user. Never "guess and proceed" on an unclear requirement.
3. **This command only distills, not executes**: Output is a Task list. Actual execution goes to /新功能 or /快速修复. Do not start implementation from within this command.
4. **客户沟通师 output is read-only**: The semantic enhancement from client-agent must be passed to the user as-is. Main process must not rewrite or summarize it away.
5. **User confirmation before handoff**: Before completing, display the original requirement → distilled Task mapping and ask if adjustment is needed. Do not silently proceed.
</content>
</section>

<section id="scope">
<title>本 command 覆盖范围</title>
<content>
**触发场景**：新需求进来 / 客户发来长段描述 / 团队对齐启动某个功能

**不适用**：已有明确 Task → 直接 /新功能 或 /快速修复 执行

**产出**：结构化 Task 列表（ID / 标题 / 业务目标 / 验收标准 / 优先级 / 推荐 Agent）
</content>
</section>

<section id="dispatch-flow">
<title>调度流水线</title>
<content>
**Step 1 — 原始需求接收**（主进程执行）
- 前置：`$ARGUMENTS` 不为空，否则 STOP
- 原样记录 `$ARGUMENTS`，作为可追溯的原始输入

**Step 2 — 语义增强**（派 客户沟通师 Agent）
- 目的：用户语言常混杂"真实想要什么" vs "他以为的解决方案"
- 产出：
  - 真实业务目标
  - 显式要求清单
  - 隐含假设清单（需向用户确认）
  - 边界问题（In-scope / Out-of-scope 疑问）
  - 紧急度 / 质量标准 / 预算信号

**Step 3 — 歧义澄清**（条件：client 识别出歧义，主进程暂停等用户）
- 主进程列出澄清问题，等用户拍板
- 规则：不得"猜着开派"——歧义就是停

**Step 4 — Task 拆分**（派 项目管理师 Agent）
- 输入：Step 2 语义增强结果 + Step 3 澄清结论（若有）
- 产出：
  - Task 列表（ID / 标题 / 业务目标 / 验收标准 / 优先级 / 推荐 Agent）
  - 依赖关系
  - 建议执行顺序
  - 风险登记

**Step 5 — 用户确认**（主进程执行）
- 展示：原始需求 → 蒸馏后 Task 列表映射
- 展示第一个建议执行 Task
- 询问是否需要调整
</content>
</section>

</content>
