description: 只读：显示当前项目状态（活跃 Task、进度、阻塞项、最近调度记录）。供用户快速回到上下文，无任何文件写入。触发信号："项目状态"、"现在做到哪了"、"有什么阻塞"。

<command>

<section id="rules">
<title>铁律</title>
<content>
1. **Read-only, zero file writes**: This command MUST NOT write, edit, or delete any file. It is purely observational.
2. **Must be in a project**: If not inside a project directory (no .git or equivalent), refuse and prompt user to cd into project root or run /项目分析.
3. **Output stays under 1 screen**: Do not dump entire TASK.md or progress-log.md. Summarize. If Tasks > 10, show Top 5 in-progress + Top 5 waiting.
4. **Proactively surface risks**: Tasks with 2+ rework rounds or 7+ days without movement must be highlighted — do not wait for user to ask.
5. **No dispatch or recommendations inside this command**: Just report current state. Do not issue dispatch suggestions or next-step recommendations (that is pm's job).
6. **No speculation**: Only report facts from files. Do not infer or guess about task status.
7. **Timestamp all data**: Every reported item must include its last-update timestamp so user knows how stale the information is.
</content>
</section>

<section id="role">
<title>角色定位</title>
<content>
本 command 是 Harness 体系的"仪表盘"。它不做任何改变，只提供快照——让用户在任何时候都能一眼看清"现在做到哪了、有什么阻塞、哪些 Task 有风险"。

核心价值：降低"上次做到哪了"的认知成本，让用户快速回到工作上下文，识别需要优先关注的阻塞和风险。

Unlike `/进度同步`：本 command 只读取不写入。Unlike `/项目分析`：本 command 不生成 CLAUDE.md，只读取现有状态文件。Unlike `@pm`：本 command 不做调度决策，只呈现事实。
</content>
</section>

<section id="scope">
<title>职责边界（In-Scope）</title>
<content>
**本 command 负责的范围**：
1. 读取并汇总项目状态信息源（TASK.md / progress-log.md / CLAUDE.md 最近进度段）
2. 展示活跃 Task 列表（进行中 + 待开始）
3. 展示最近完成的 Task（近 5 条）
4. 展示最近调度记录（progress-log 近 5 条）
5. 主动标记风险 Task（≥2 轮返工 / ≥7 天无动作）
6. 展示当前项目模式（poc / production / debug / weak-model）
7. 展示阻塞项清单
</content>
</section>

<section id="not-scope">
<title>不承担的职责（Out-of-Scope）</title>
<content>
1. **修改任何文件**：本 command 是纯只读，不写入 → 需写入时调用 `/进度同步`
2. **调度建议或下一步推荐**：不决定"接下来该做什么" → 这是 `@pm` 的职责
3. **创建缺失的状态文件**：如果 TASK.md 不存在，提示用户初始化，但不自动创建
4. **跨项目状态汇总**：只展示当前项目，不汇总其他项目
5. **详细代码审查结果**：不展示代码层面的具体问题 → 查看 `@code-review` 报告
6. **历史考古**：不追溯超过 30 天的记录 → 需查历史请读原始文件
</content>
</section>

<section id="skill-tree">
<title>技能树</title>
<content>
**一级：状态信息聚合**
- 二级：多源数据读取（TASK.md / progress-log.md / CLAUDE.md）
  - 三级：Task 状态解析、调度记录提取、最近进度段定位、时间戳识别
- 二级：信息去重与冲突检测（同一 Task 在多文件中的状态不一致）
  - 三级：状态优先级规则（TASK.md > progress-log > CLAUDE.md）、冲突标记

**一级：风险识别**
- 二级：停滞检测（无动作天数 / 无调度记录天数）
  - 三级：时间戳分析、最后活动识别、自动停滞标记
- 二级：返工检测（同一 Task 被多次 dispatch 到同一 Agent）
  - 三级：dispatch 记录解析、返工次数统计、返工原因分类

**一级：信息呈现**
- 二级：摘要生成（将多源信息压缩为 1 屏幕可读内容）
  - 三级：优先级排序、Top-N 筛选、关键信息突出
- 二级：格式化输出（统一模板、清晰层级、时间戳标注）
  - 三级：Markdown 表格、状态标签、风险高亮
</content>
</section>

<section id="methodology">
<title>方法论与执行步骤</title>
<content>
**Workflow — 3 步流水线**

**Step 1 — 项目识别**（主进程执行）
- 确定当前工作目录对应项目
- 检查是否存在项目标记（`.git` / `package.json` 等）
- 若不在任何项目内 → STOP，提示用户 cd 到项目或先跑 `/项目分析`

**Step 2 — 数据采集**（主进程执行，只读）
- 读取 `tasks/TASK.md`：提取所有 Task 的 ID / 标题 / 状态 / 负责 Agent / 阻塞信息
- 读取 `progress-log.md` 最近 30 行：提取最近调度记录
- 读取项目根 `CLAUDE.md` 的 `## 最近进度` 段：提取最近 5 条进度
- 检查各层 CLAUDE.md 的最近更新时间（判断信息新鲜度）

**Step 3 — 状态报告输出**（主进程执行）
- 按模板格式化输出（见 output-contract）
- 计算并标记风险：
  - 返工风险：同一 Task 被 dispatch 到同一 Agent ≥ 2 次
  - 停滞风险：Task 状态为 in-progress 但 ≥ 7 天无调度记录
  - 阻塞风险：Task 标注了 blockedBy 或阻塞说明
- 若无 TASK.md → 提示"项目尚未初始化 Task 文件，建议运行 /需求蒸馏"

**关键决策点**：
- 信息源冲突（TASK.md 说进行中，progress-log 说已完成）→ 标记冲突，以 TASK.md 为准
- Task 数量过多（>10）→ 只展示 Top 5 进行中和 Top 5 待开始
- 无项目标记 → 拒绝执行

**常见反模式与对策**：
| 反模式 | 表现 | 对策 |
|--------|------|------|
| 信息倾倒 | 把整个 TASK.md 都输出 | 铁律 3：压缩到 1 屏幕，只展示 Top-N |
| 过时信息 | 展示了很久没更新的 Task 但不说多旧 | 铁律 7：每条信息带时间戳 |
| 越权推荐 | 告诉用户"接下来该派谁" | 铁律 5：不做调度建议，只报告事实 |
| 猜测状态 | "这个 Task 可能快完成了" | 铁律 6：只报告文件中明确记录的状态 |
</content>
</section>

<section id="collaboration">
<title>协作边界</title>
<content>
**上游（谁触发本 command）**：
- 用户直接输入 `/项目状态`
- 用户输入："现在做到哪了"、"有什么阻塞"、"进度如何"
- 会话开始时快速了解上下文

**下游（本 command 派谁）**：
- 本 command 不派 Agent，由主进程直接执行

**横向衔接**：
- 与 `/进度同步`：本 command 读取的信息源之一是 `/进度同步` 写入的内容
- 与 `/项目分析`：本 command 读取 CLAUDE.md，而 `/项目分析` 创建/更新 CLAUDE.md
- 与 `@pm`：本 command 呈现状态，`@pm` 基于状态做调度决策
</content>
</section>

<section id="dispatch-signals">
<title>调度触发信号</title>
<content>

- 用户输入："/项目状态"、"现在做到哪了"、"有什么阻塞"、"进度如何"

- 想记录进度 → `/进度同步`
