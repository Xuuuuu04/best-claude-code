---
description: 将当前会话上下文、进度、决策、变更同步写入项目各层 CLAUDE.md 的"最近进度"段，让下次会话任意层级打开都能回到上下文。触发信号：阶段性工作完成、要下班前、会话 context 接近饱和、完成关键决策。
argument-hint: [可选：本次进度的关键摘要，将附加到进度记录]
---

<command>

<section id="rules">
<title>铁律</title>
<content>
1. **Only write to project CLAUDE.md files**: Hook-A blocks writes to `~/.claude/` core files. This command only updates CLAUDE.md files within the current project directory.
2. **Append only, never overwrite**: Only append to the `## 最近进度` section. Never delete or overwrite existing content in CLAUDE.md.
3. **Existing CLAUDE.md only**: Do not create new CLAUDE.md files. If a directory has no CLAUDE.md, skip it — use /项目分析 first.
4. **No debug/temp info**: Progress log entries must contain meaningful information (decisions, changes, blockers, todos) — no temporary debugging notes.
5. **Single entry ≤ 15 lines**: Each progress entry must stay concise. If more detail is needed, link to a separate document.
6. **Preserve user-content markers**: Content inside `<!-- user-content -->` markers must be preserved unchanged.
7. **Layer appropriately**: Root CLAUDE.md gets cross-module impact; module CLAUDE.md gets local changes; sub-module CLAUDE.md gets file-level details.
</content>
</section>

<section id="role">
<title>角色定位</title>
<content>
本 command 是 Harness 体系的"上下文保鲜器"。在长时间的开发过程中，会话上下文会丢失、人员会切换、记忆会模糊——本 command 通过将关键进度和决策写入项目各层的 CLAUDE.md，确保任何人、任何时间打开项目的任何目录，都能快速回到上下文。

核心价值：降低"上次做到哪了"的认知成本，让项目状态对所有人可见，让历史决策可追溯。

Unlike `/项目状态`：本 command 写入状态，而 `/项目状态` 只读取。Unlike `/项目分析`：本 command 更新已有 CLAUDE.md 的进度段，而 `/项目分析` 是全面重建所有 CLAUDE.md。Unlike `@pm` 的 progress-log：本 command 将进度写入项目目录结构，而 progress-log 是集中式日志。
</content>
</section>

<section id="scope">
<title>职责边界（In-Scope）</title>
<content>
**本 command 负责的范围**：
1. 从当前会话提取关键上下文（已完成变更 / 架构决策 / 当前进度 / 待办阻塞 / 已知问题）
2. 定位项目内所有已存在的 CLAUDE.md（排除 `~/.claude/` 核心目录）
3. 按分层策略将进度信息追加到各层 CLAUDE.md 的 `## 最近进度` 段
4. 条件性同步长期价值信息到 Memory（`~/.claude/projects/{project}/memory/`）
5. 维护条目保留规则（模块级 ≤ 10 条，根级 ≤ 20 条，超出归档）
6. 产出同步摘要报告
</content>
</section>

<section id="not-scope">
<title>不承担的职责（Out-of-Scope）</title>
<content>
1. **创建新的 CLAUDE.md**：只更新已存在的文件，新建文件属于 `/项目分析` 范畴
2. **修改 CLAUDE.md 的非进度段**：如"技术栈"、"架构"等固定内容段不在本 command 范围内
3. **写入 `~/.claude/` 核心目录**：Hook-A 物理拦截，违反铁律 1
4. **替代 `@pm` 的 Task 管理**：本 command 不管理 Task 生命周期，只记录进度到文档
5. **替代 Git commit message**：代码变更的历史记录由 Git 管理，本 command 记录的是"为什么这样改"的决策上下文
6. **实时持续同步**：本 command 是显式触发的一次性动作，不是自动持续同步
</content>
</section>

<section id="skill-tree">
<title>技能树</title>
<content>
**一级：上下文提取**
- 二级：会话信息萃取（变更 / 决策 / 进度 / 阻塞 / 问题）
  - 三级：文件变更列表提取、决策 rationale 识别、阻塞项分类、风险分级
- 二级：信息价值判断（什么值得写入 / 什么是临时噪音）
  - 三级：调试信息过滤、临时方案标记、长期价值识别

**一级：分层写入策略**
- 二级：内容分层映射（根 / 模块 / 子模块 的内容粒度区分）
  - 三级：跨模块影响判断、模块内变更聚合、文件级细节下沉
- 二级：格式标准化（时间戳 / 标题 / 字段 / 关联链接）
  - 三级：Markdown 模板填充、YAML frontmatter 保护、user-content 标记识别

**一级：归档与清理**
- 二级：条目数量控制（模块级 ≤ 10 / 根级 ≤ 20）
  - 三级：超量检测、自动归档到 progress-archive.md、归档策略（最早/最低价值）
- 二级：Memory 同步（长期价值信息入 Memory）
  - 三级：信息类型判断（user / feedback / project / reference）、MEMORY.md 索引更新
</content>

<section id="methodology">
<title>方法论与执行步骤</title>
<content>
**Workflow — 5 步流水线**

**Step 1 — 收集上下文**（主进程执行）
- 从当前会话提取 6 个维度：
  1. **已完成的变更**：文件增/删/改列表，按模块分组
  2. **架构决策**：做了什么决定、为什么（rationale）、否决了哪些备选
  3. **当前任务进度**：正在做什么、完成到哪、下一步是什么
  4. **待办/阻塞**：剩余步骤清单、被什么阻塞、需要什么资源
  5. **已知问题**：发现的 bug / 技术债 / 风险 / 需关注事项
  6. **关键上下文**：后续开发必须知道的信息（如临时方案、约定变更）
- 将 `$ARGUMENTS` 作为用户提供的额外摘要补充

**Step 2 — 定位所有 CLAUDE.md**（主进程执行）
- Glob 扫描项目内所有 `CLAUDE.md`（排除 `~/.claude/` 下的）
- 记录每个 CLAUDE.md 的目录层级（根 / 模块 / 子模块）
- 检查每个文件是否已有 `## 最近进度` 段，没有则创建

**Step 3 — 分层适配写入**（主进程执行）
- **根目录层 CLAUDE.md**：写入跨模块影响的内容
  - 当前里程碑 / 全局架构变更 / 跨模块影响 / 全局待办与风险 / 重大决策
- **模块级 CLAUDE.md**：写入本模块相关的内容
  - 本模块文件变更 / 接口变更 / 模块待办 / 相关架构决策
- **子模块级 CLAUDE.md**：写入最具体的内容
  - 具体文件变更 / 函数/类级修改 / 本子模块 bug 修复 / 实现细节

- 写入格式（统一模板，追加到 `## 最近进度` 段顶部，时间倒序）：
```markdown
### {YYYY-MM-DD HH:mm} — {简短标题}
- **变更类型**：{新增 / 修复 / 重构 / 配置 / 文档}
- **影响范围**：{本模块 / 跨模块 - 哪些模块}
- **变更内容**：
  - {具体变更项 1}
  - {具体变更项 2}
- **架构决策**：{如有 rationale，一句话概括}
- **待办**：{如未完成，列出剩余步骤}
- **关联**：{Task ID / Issue / PR}
```

**Step 4 — 条目保留与归档**（主进程执行）
- 检查每层 `## 最近进度` 的条目数量
- 模块/子模块级：> 10 条时，将最早/价值最低的条目移到同目录 `progress-archive.md`
- 根目录级：> 20 条时，归档到 `docs/progress-archive.md`

**Step 5 — Memory 同步**（主进程执行，条件触发）
- 判断本次信息是否有长期价值（如架构决策、重要约定、已知风险）
- 如有，写入 `~/.claude/projects/{project}/memory/`（Hook-A 白名单允许）
- 更新 `MEMORY.md` 索引

**Step 6 — 汇报**
- 输出同步摘要：更新了 N 个 CLAUDE.md / 根目录摘要 / Memory 同步结果

**关键决策点**：
- 某目录无 CLAUDE.md → 跳过，不创建（避免污染项目结构）
- 信息粒度过大 → 分层下沉，根目录只保留跨模块摘要
- 发现无长期价值的信息 → 不写入 Memory

**常见反模式与对策**：
| 反模式 | 表现 | 对策 |
|--------|------|------|
| 信息倾倒 | 把会话所有内容都写进去 | 铁律 4：只写有意义的信息 |
| 单层重复 | 根目录和子模块写一样的内容 | 分层策略：根目录跨模块，子模块文件级 |
| 覆盖历史 | 新进度替换旧进度 | 铁律 2：append only，never overwrite |
| 无关联上下文 | 写了变更但没写为什么 | 模板强制包含"架构决策"字段 |
</content>
</section>

<section id="collaboration">
<title>协作边界</title>
<content>
**上游（谁触发本 command）**：
- 用户直接输入 `/进度同步 [可选摘要]`
- `@pm` 在阶段性里程碑完成后触发
- 会话 context 接近饱和时自动提醒用户使用

**下游（本 command 派谁）**：
- 本 command 不派 Agent，由主进程直接执行
- 若发现项目无 CLAUDE.md → 建议用户先运行 `/项目分析`

**横向衔接**：
- 与 `/项目分析`：`/项目分析` 创建 CLAUDE.md，本 command 更新其进度段
- 与 `/项目状态`：本 command 写入的进度是 `/项目状态` 读取的信息源之一
- 与 `@pm` 的 progress-log：progress-log 是集中式调度日志，本 command 将关键信息分发到项目目录结构中
</content>
</section>

<section id="dispatch-signals">
<title>调度触发信号</title>
<content>
**主进程应触发本 command 的场景**：
- 用户输入包含："进度同步"、"保存进度"、"记录一下"、"上下文同步"
- 阶段性工作完成（如一个 Task 做完、一个 Phase 结束）
- 会话 context 接近饱和（Claude 提示"对话较长"时）
- 完成关键架构决策或重大变更后
- 用户要下班前/会话结束前

**主进程不应触发本 command 的场景（防误派）**：
- 想了解当前状态但不修改 → `/项目状态`
- 项目尚未初始化 CLAUDE.md → `/项目分析`
- 仅想查看 Task 列表 → 直接读 `tasks/TASK.md`
</content>
</section>
