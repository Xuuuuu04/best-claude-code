<section id="rules">
<title>铁律</title>
<content>
1. **No business code changes**: This command ONLY touches file organization — moves, renames, and deletions of non-code structural files. Any change that alters import paths, breaks builds, or modifies logic MUST be stopped and escalated to dev-lead.
2. **User approval before any move/delete**: The dispatch flow MUST pause after producing the reorganization plan and wait for explicit user approval before any file operations are executed.
3. **Backup or git-clean precondition**: Before executing any move or delete, verify that `git status` is clean or that the user has acknowledged backup responsibility. No irreversible file operations without this check.
4. **Scope limited to structural files**: Directories like `node_modules` / `.git` / `dist` / `build` / `__pycache__` are excluded from analysis. Config files that affect build behavior require dev-lead sign-off.
5. **STOP if import paths break**: If moving a file would require updating import paths in business code, STOP — escalate to dev-lead to handle it as part of /新功能 or /快速修复.
6. **Never delete without identification**: Every deletion MUST identify what the file is, why it's redundant, and what confirms it's safe to delete. "Looks old" is not sufficient.
7. **Document every operation**: Every move/rename/delete MUST be logged with: original path → new path (or DELETED) → reason. Silent operations are forbidden.
</content>
</section>

<section id="role">
<title>角色定位</title>
<content>
本 command 是项目结构卫生的维护者。它不负责架构决策、不负责代码质量、不负责功能实现——它只做一件事：让文件系统的组织方式反映项目的真实结构，移除无价值的噪音。

核心价值：通过系统化的目录扫描和整理，减少开发者在"找文件"上的认知负担，清除历史遗留的冗余和混乱。这是一个低风险、高频率的维护动作，但必须在严格的边界内进行，防止"整理目录"变成"破坏构建"。

Unlike `/架构重构`：本 command 不改变模块职责或接口，只改变文件位置。Unlike `/新功能`：本 command 不添加任何新代码。Unlike `@dev-lead`：本 command 不产出技术方案，只执行结构整理。
</content>
</section>

<section id="scope">
<title>职责边界（In-Scope）</title>
<content>
**本 command 负责的范围**：
1. 冗余文件识别与清理（重复内容 / 空文件 / 临时文件如 `*.tmp` / `*.bak` / `test-*.log` / `.DS_Store` 累积）
2. 重复目录识别与合并（相同职责的目录存在多个副本，如 `utils/` 和 `utilities/`）
3. 命名规范化（大小写混用 / 空格 / 中文路径 / 不一致的命名风格，如 camelCase vs kebab-case）
4. 孤儿文件识别（不被任何代码引用且不属于文档/配置的文件）
5. 过深嵌套治理（超过 4 层无业务价值的目录嵌套）
6. 空目录清理（无任何内容的空文件夹）
7. 产出整理报告和操作日志
</content>
</section>

<section id="not-scope">
<title>不承担的职责（Out-of-Scope）</title>
<content>
1. **业务代码文件的内容修改**：只移动/重命名/删除，不修改文件内容 → 需修改内容时派 `@backend` / `@frontend`
2. **影响 import 路径的移动**：如果移动会导致 import/require 路径断裂 → STOP，派 `@dev-lead` 处理
3. **`.env` / 密钥文件 / 安全敏感配置**：这些文件的任何操作需要 `@security-auditor` 评审
4. **用户明确标注"不动"的目录**：尊重用户的排除标记
5. **构建配置文件的结构性修改**：如 `vite.config.ts` / `webpack.config.js` 的目录 alias 变更 → 派 `@dev-lead`
6. **Git 子模块 / LFS 对象的管理**：Git 内部机制不在本 command 范围内
7. **多项目之间的目录合并**：跨仓库的目录重组属于 `/架构重构` 范畴
</content>
</section>

<section id="skill-tree">
<title>技能树</title>
<content>
**一级：目录结构分析**
- 二级：文件系统扫描（Glob 模式 / 排除规则 / 深度限制）
  - 三级：`find` / `Glob` 扫描、排除清单配置、嵌套深度检测
- 二级：冗余识别算法（内容哈希比对 / 文件名模式匹配 / 时间戳分析）
  - 三级：MD5/sha256 内容比对、`*.{tmp,bak,log}` 模式识别、长期未访问检测

**一级：命名规范治理**
- 二级：命名风格统一（camelCase / PascalCase / kebab-case / snake_case）
  - 三级：项目约定提取、不一致检测、批量重命名方案设计
- 二级：路径合法性校验（非法字符 / 空格 / 中文 / 过长路径）
  - 三级：跨平台兼容性检查（Windows 限制 / macOS 大小写敏感 / Linux 路径长度）

**一级：安全执行**
- 二级：操作前校验（git status / 备份确认 / 影响面评估）
  - 三级：干净工作区检测、stash 建议、import 路径影响预判
- 二级：操作后验证（文件存在性 / 路径可访问性 / 构建完整性）
  - 三级：快速构建测试、关键路径检查、回滚准备
</content>

<section id="methodology">
<title>方法论与执行步骤</title>
<content>
**Workflow — 5 步流水线**

**Step 1 — 扫描分析**（主进程执行）
- 确定 target = `$ARGUMENTS` 或当前工作目录
- 扫描目标目录树，排除：`node_modules` / `.git` / `dist` / `build` / `__pycache__` / `.next` / `target` / `.venv` / `coverage`
- 识别问题清单，按类别分组：
  - 冗余文件：重复内容 / 空文件 / 临时文件 / 日志文件
  - 重复目录：相同职责的多目录副本
  - 命名不规范：大小写混用 / 含空格 / 中文路径 / 风格不一致
  - 孤儿文件：无引用、非文档/配置的文件
  - 过深嵌套：> 4 层且无明显业务分组价值
  - 空目录：无任何内容的文件夹
- 每项标注：路径 / 问题类型 / 严重度（高：影响开发体验 / 中：命名不规范 / 低：历史遗留）
- 输出：扫描报告（问题总数 + 分类统计）

**Step 2 — 整理方案设计**（主进程执行）
- 基于 Step 1 问题报告，产出具体整理方案
- 每项操作明确：操作类型（移动/重命名/删除）+ 原路径 + 目标路径 + 理由
- 评估每项操作是否影响 import 路径（若影响 → 标记为"需 dev-lead 额外处理"，不纳入本 command 执行范围）
- 产出整理方案文件：`docs/directory-cleanup-plan.md`

**Step 3 — 用户拍板（强制停顿）**
- 主进程呈现整理方案摘要：
  - 影响文件数 / 操作类型分布（移动 X / 重命名 Y / 删除 Z）
  - 高严重度问题清单（建议优先处理）
  - 标记为"需 dev-lead"的项（不在本次执行范围内）
- **必须等待用户明确确认才进入 Step 4**
- 用户可要求排除某些项目或调整优先级

**Step 4 — 执行整理**（主进程执行）
- 预执行检查：`git status` 是否干净？→ 不干净则提示用户先 commit/stash
- 按 Step 2 批准方案执行：移动 / 重命名 / 删除（仅安全操作）
- 每项操作后输出结果日志（原路径 → 新路径 / DELETED + 结果）
- 遇到任何标记外的影响 → 立即 STOP，记录并汇报

**Step 5 — 验证无破坏**（主进程执行）
- 验证整理后项目构建不破坏（尝试 `npm run build` / `cargo build` / `go build` / `python -m compileall` 等，如存在构建脚本）
- 验证无意外删除（`git status` 或 `git diff` 确认改动范围符合预期）
- 产出验证报告

**关键决策点**：
- git 工作区不干净 → 提示用户先 commit/stash，不强制执行
- 发现影响 import 路径的操作 → 标记为 dev-lead 项，不执行
- 用户未确认方案 → STOP，等待确认

**常见反模式与对策**：
| 反模式 | 表现 | 对策 |
|--------|------|------|
| 过度清理 | 删除仍在使用的文件 | 每项删除必须有"不被引用"证据 |
| 盲目重命名 | 重命名破坏 import | 预判影响，影响 import 的标记为 dev-lead 项 |
| 无日志操作 | 执行了但用户不知道改了什么 | 强制记录每项操作的 before/after |
| 递归灾难 | 误删整个目录树 | 删除前确认目录内容，空目录才允许批量删除 |
</content>

<section id="collaboration">
<title>协作边界</title>
<content>
**上游（谁触发本 command）**：
- 用户直接输入 `/目录整理 [可选路径]`
- 或 `@pm` 在项目维护阶段识别出目录结构问题

**下游（本 command 派谁）**：
- 本 command 不派 Agent，由主进程直接执行（纯编排命令）
- 若发现影响 import 路径的项 → 建议用户后续派 `@dev-lead` 处理
- 若发现安全敏感文件需处理 → 建议用户派 `@security-auditor` 评审

**横向衔接**：
- 与 `/项目分析`：整理完成后可调用 `/项目分析` 更新 CLAUDE.md 的文件索引
- 与 `/进度同步`：将整理结果记录到各层 CLAUDE.md 的"最近进度"
- 与 `/架构重构`：若整理过程中发现目录结构问题根因是架构设计缺陷 → 建议用户考虑 `/架构重构`
</content>

<title>Final Reminder</title>
