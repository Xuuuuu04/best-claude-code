<section id="rules">
<title>铁律</title>
<content>
1. **Architect leads, always**: Architecture refactoring MUST be initiated by the architect Agent. Main process may not design the refactoring plan. Any "shortcut" that bypasses architect = invalid.
2. **User approval is a hard gate**: The flow MUST stop after architect delivers the ADR and refactoring plan. No code changes happen until the user explicitly approves the plan. This gate CANNOT be skipped for any reason.
3. **ADR before any code**: An Architecture Decision Record (ADR) documenting the why, what, tradeoffs, and rollback strategy is MANDATORY before any implementation begins.
4. **Quality gates are non-negotiable**: code-review + security-auditor + test-func + test-lead are all mandatory. No conditional skips allowed on architecture-level changes.
5. **Rollback plan is mandatory**: Every refactoring plan MUST include a rollback strategy. If no rollback is possible, that MUST be explicitly stated and user must re-confirm.
</content>
</section>

<section id="scope">
<title>本 command 覆盖范围</title>
<content>
**适用场景**
- 模块边界重划（单体拆微服务 / 微服务合并 / 模块职责重分配）
- 通信协议改造（REST → gRPC / 同步 → 异步消息队列）
- 基础设施更换（数据库迁移 / 消息中间件替换 / 缓存层重构）
- 跨模块数据模型重构（影响超过 3 个模块的表结构变更）
- 核心框架升级（框架大版本迁移 / 运行时替换）

**不适用场景**（用其他命令）
- 单文件/单模块内重构 → 直接派 开发组长 + 对应开发 Agent
- bug 修复 → /快速修复
- 目录和文件结构整理 → /目录整理
- 新功能添加 → /新功能
</content>
</section>

<section id="dispatch-flow">
<title>调度流水线</title>
<content>
**Step 1 — 现状诊断**（派 架构师 Agent）
- 读取现有架构文档（CLAUDE.md / ADR / 架构图）
- 识别当前架构的痛点和重构必要性
- 评估重构复杂度（影响模块数 / 数据迁移风险 / 停机时间）
- 输入：`$ARGUMENTS`（用户描述的重构目标）

**Step 2 — ADR + 重构方案**（architect 继续）
- 产出 ADR（Architecture Decision Record）：
  - 背景与问题
  - 决策内容
  - 方案对比（至少 2 个候选 + 选择理由）
  - 影响与权衡
  - 回滚策略（若无可回滚方案须明确说明）
- 产出分阶段实施计划（Phase 划分 + 每 Phase 的验收标准）
- 归档：`docs/adr/ADR-{YYYY-MM-DD}-{title}.md`

**Step 3 — 用户拍板（强制停顿，不可跳过）**
- 主进程向用户呈现：ADR 摘要 + 影响范围 + 风险 + 回滚策略
- **明确停止，等待用户书面确认**
- 用户可要求修改方案或中止
- 未收到确认前禁止任何代码改动

**Step 4 — 实现方案拆解**（派 开发组长 Agent）
- 基于 architect 的 ADR，翻译为文件级实现方案
- 按 Phase 拆解为可独立执行的 Task 列表
- 识别各模块的改动范围和依赖顺序

**Step 5 — 分模块实现**（按 Phase 逐步派开发 Agent）
- 每 Phase 串行执行（不并行）
- 根据涉及技术选派：backend / frontend / database（各自独立 Task）
- 每个 Agent 完成后推荐 code-review 再进入下一步

**Step 6 — 代码审查**（派 代码审计师 Agent，每 Phase 必须）
- 审查变更是否符合 ADR 中的架构决策
- 识别引入的新技术债

**Step 7 — 安全审计**（派 安全审计师 Agent，必须）
- 架构变更对认证 / 授权 / 数据流的安全影响
- 新引入的技术栈和依赖 CVE 扫描

**Step 8 — 功能测试**（派 功能测试师 Agent，必须）
- 端到端用户流程回归测试
- 重构前后行为一致性验证

**Step 9 — 测试总监师裁决**（必须，最终闸门）
- 综合所有报告做最终验收裁决
- 裁决为"通过"才允许合并和部署

**Step 10 — 归档**
- 更新项目架构文档 CLAUDE.md
- 归档 ADR 到 `docs/adr/`
- progress-log.md 记录重构完成里程碑
</content>

<section id="final_reminder">
<title>Final Reminder</title>
<content>
- Architect leads — no one else designs the refactoring plan.
- ADR is mandatory before any code. No ADR = no execution.
- User approval at Step 3 is a hard gate. No bypass for any reason.
- All 4 quality gates (code-review + security-auditor + test-func + test-lead) are mandatory. No conditional skips.
- Rollback strategy must exist. If it doesn't, user must explicitly re-acknowledge the risk.
- Each Phase is serial — never start Phase N+1 while Phase N has unresolved issues.
