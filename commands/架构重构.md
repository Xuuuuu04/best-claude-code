<section id="rules">
<title>铁律</title>
<content>
1. **Architect leads, always**: Architecture refactoring MUST be initiated by the architect Agent. Main process may not design the refactoring plan. Any "shortcut" that bypasses architect = invalid.
2. **User approval is a hard gate**: The flow MUST stop after architect delivers the ADR and refactoring plan. No code changes happen until the user explicitly approves the plan. This gate CANNOT be skipped for any reason.
3. **ADR before any code**: An Architecture Decision Record (ADR) documenting the why, what, tradeoffs, and rollback strategy is MANDATORY before any implementation begins.
4. **Quality gates are non-negotiable**: code-review + security-auditor + test-func + test-lead are all mandatory. No conditional skips allowed on architecture-level changes.
5. **Rollback plan is mandatory**: Every refactoring plan MUST include a rollback strategy. If no rollback is possible, that MUST be explicitly stated and user must re-confirm.
6. **Phase serial, no overlap**: Each Phase is serial — never start Phase N+1 while Phase N has unresolved issues or unmerged changes.
7. **Never refactor without baseline**: Current system state (tests passing, metrics baseline) must be documented before any change. No baseline = no start.
</content>
</section>

<section id="role">
<title>角色定位</title>
<content>
本 command 是 Harness 体系中架构级变更的统一入口。它不是某个专职 Agent，而是一套跨多角色协作的编排流水线， orchestrating `@architect` → `@dev-lead` → implementation agents → `@code-review` → `@security-auditor` → `@test-func` → `@test-lead` 的完整链路。

核心价值：在架构变更的重度不确定性中，通过强制 ADR 文档化、强制用户拍板、强制质量闭环，将"拍脑袋重构"转化为"可追溯、可回滚、可验证"的工程实践。

Unlike `/新功能`：本 command 不添加新能力，而是改变现有系统的结构。Unlike `/目录整理`：本 command 改变模块职责和数据流，而非仅整理文件位置。Unlike `/快速修复`：本 command 是计划性工程，不是缺陷响应。
</content>
</section>

<section id="scope">
<title>职责边界（In-Scope）</title>
<content>
**本 command 负责的范围**：
1. 模块边界重划（单体拆微服务 / 微服务合并 / 模块职责重分配）
2. 通信协议改造（REST → gRPC / 同步 → 异步消息队列 / 新增事件总线）
3. 基础设施更换（数据库迁移 / 消息中间件替换 / 缓存层重构 / 搜索引擎替换）
4. 跨模块数据模型重构（影响超过 3 个模块的表结构变更）
5. 核心框架升级（框架大版本迁移 / 运行时替换 / 语言版本升级）
6. 架构治理文档产出（ADR / 架构图更新 / 迁移手册）
7. 质量闭环执行（代码审查 → 安全审计 → 功能测试 → 总监裁决）
</content>
</section>

<section id="not-scope">
<title>不承担的职责（Out-of-Scope）</title>
<content>
1. **单文件/单模块内重构**：不改变模块边界、不新增服务、不影响外部接口的局部重构 → 直接派 `@dev-lead` + 对应开发 Agent，无需本 command
2. **Bug 修复**：现有功能缺陷的修复 → `/快速修复`
3. **目录和文件结构整理**：仅移动/重命名文件，不改变模块职责或接口 → `/目录整理`
4. **新功能添加**：在现有架构上增加新能力 → `/新功能`
5. **纯依赖升级**：仅升级依赖版本，不改变架构 → 派 `@backend` 或 `@devops` 处理
6. **数据库迁移的执行细节**：DDL/DML 脚本编写属于 `@database` 职责，本 command 只定策略
</content>
</section>

<section id="methodology">
<title>方法论与执行步骤</title>
<content>
**Workflow — 10 步流水线（全部串行，不可并行）**

**Step 1 — 现状诊断**（派 `@architect` Agent）
- 读取现有架构文档（CLAUDE.md / ADR / 架构图）
- 识别当前架构痛点和重构必要性
- 评估重构复杂度：影响模块数 / 数据迁移风险 / 停机时间 / 团队认知成本
- 输入：`$ARGUMENTS`（用户描述的重构目标）
- 输出：现状诊断报告，含复杂度评级（低/中/高/极高）

**Step 2 — ADR + 重构方案**（`@architect` 继续）
- 产出 ADR（Architecture Decision Record）：
  - 背景与问题（为什么必须重构）
  - 决策内容（具体做什么）
  - 方案对比（至少 2 个候选 + 量化对比维度 + 选择理由）
  - 影响与权衡（正面 / 负面 / 风险）
  - 回滚策略（若无可回滚方案须明确说明 + 用户必须额外确认）
- 产出分阶段实施计划：Phase 划分 + 每 Phase 的验收标准 + 依赖顺序
- 归档：`docs/adr/ADR-{YYYY-MM-DD}-{title}.md`

**Step 3 — 用户拍板（强制停顿，不可跳过）**
- 主进程向用户呈现：ADR 摘要 / 影响范围 / 风险 / 回滚策略 / Phase 计划
- **明确停止，等待用户书面确认**
- 用户可要求修改方案、增加 Phase、或中止
- 未收到确认前禁止任何代码改动

**Step 4 — 实现方案拆解**（派 `@dev-lead` Agent）
- 基于 architect 的 ADR，翻译为文件级实现方案
- 按 Phase 拆解为可独立执行的 Task 列表（每个 Task 含：文件路径 + 具体变更 + 验收标准）
- 识别各模块的改动范围和依赖顺序
- 标记需 `@database` 介入的 Schema 变更项

**Step 5 — 分模块实现**（按 Phase 逐步派开发 Agent）
- 每 Phase 串行执行（不并行，防止交叉污染）
- 根据涉及技术选派：`@backend` / `@frontend` / `@database`（各自独立 Task）
- 每个 Agent 完成后必须通过 `@code-review` 再进入下一步

**Step 6 — 代码审查**（派 `@code-review` Agent，每 Phase 必须）
- 审查变更是否符合 ADR 中的架构决策
- 识别引入的新技术债和架构漂移
- 产出审查报告， verdict：APPROVED / CHANGES REQUESTED

**Step 7 — 安全审计**（派 `@security-auditor` Agent，必须）
- 架构变更对认证 / 授权 / 数据流的安全影响
- 新引入的技术栈和依赖 CVE 扫描
- 数据迁移过程的安全风险（数据泄露 / 中间状态一致性）

**Step 8 — 功能测试**（派 `@test-func` Agent，必须）
- 端到端用户流程回归测试
- 重构前后行为一致性验证（关键路径对比）
- 性能基线对比（重构前后延迟 / 吞吐量）

**Step 9 — 测试总监师裁决**（派 `@test-lead`，必须，最终闸门）
- 综合所有报告做最终验收裁决
- 裁决为"通过"才允许合并和部署
- 裁决为"条件通过"需明确剩余风险和监控要求

**Step 10 — 归档与知识沉淀**
- 更新项目架构文档 CLAUDE.md
- 归档 ADR 到 `docs/adr/`
- 更新架构图和接口文档
- `progress-log.md` 记录重构完成里程碑

**关键决策点**：
- Step 3 用户不批准 → 中止，不产生任何代码变更
- 复杂度评级"极高" → 建议拆分为多个独立 command 或延长周期
- 无回滚策略 → 必须用户书面确认"接受不可回滚风险"

**常见反模式与对策**：
| 反模式 | 表现 | 对策 |
|--------|------|------|
| Big Bang 重构 | 试图一次改完所有模块 | 强制分 Phase，每 Phase 独立可交付 |
| 架构漂移 | 实现偏离 ADR 决策 | 每 Phase 代码审查必须对比 ADR |
| 回滚遗忘 | 计划中无回滚方案 | ADR 中回滚策略为必填项 |
| 质量闸门跳过 | "这次改动不大，不用审了" | 铁律 4：所有质量闸门 mandatory，无条件跳过 |
</content>

<section id="collaboration">
<title>协作边界</title>
<content>
**上游（谁触发本 command）**：
- 用户直接输入 `/架构重构 [目标描述]`
- 或 `@pm` 在 Task 分解后识别出架构级变更需求，路由到本 command

**下游（本 command 派谁）**：
- Step 1-2：`@architect`（现状诊断 + ADR）
- Step 4：`@dev-lead`（方案拆解）
- Step 5：`@backend` / `@frontend` / `@database`（分模块实现）
- Step 6：`@code-review`（代码审查）
- Step 7：`@security-auditor`（安全审计）
- Step 8：`@test-func`（功能测试）
- Step 9：`@test-lead`（总监裁决）

**横向衔接**：
- 与 `/项目分析`：重构完成后调用 `/项目分析` 更新 CLAUDE.md 知识图谱
- 与 `/进度同步`：每 Phase 完成后调用 `/进度同步` 记录进度到各层 CLAUDE.md
- 与 `/目录整理`：重构涉及目录调整时，目录变更由本 command 统一处理，不单独调 `/目录整理`
</content>
</section>
