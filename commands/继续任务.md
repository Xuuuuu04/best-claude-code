<command>

<section id="rules">
<title>铁律</title>
<content>
1. **Never start new work without checking existing state**: MUST read TASK.md, progress-log.md, and relevant CLAUDE.md before any action. Blindly continuing creates divergence.
2. **Self-check before dispatch**: MUST identify what is incomplete, inconsistent, or suboptimal before dispatching any Agent. "Looks fine to me" is not sufficient.
3. **Consistency audit is mandatory**: MUST verify CLAUDE.md matches actual code, Task status matches progress-log, and scheme matches implementation. Mismatch = BLOCK + flag.
4. **Do not over-optimize**: Stop when the Task meets its DoD. Perfectionism beyond DoD wastes cycles.
5. **User visibility on findings**: Every identified gap MUST be reported to user with severity (must-fix / should-fix / nice-to-have) before dispatching fixes.
6. **One gap at a time**: Fix one category of issue per round. Do not dispatch multiple unrelated fixes in parallel.
7. **Document the delta**: After each round of improvements, append a summary to progress-log.md so the audit trail is complete.
</content>
</section>

<section id="role">
<title>角色定位</title>
<content>
本 command 是 Harness 体系的"自检推进器"。当用户说"继续"时，它不盲目执行，而是先做一次全面的健康检查——检索项目状态、审核一致性、识别未尽完善之处——然后有针对性地推进任务向最佳状态收敛。

核心价值：防止"做到80%就停了"的隐性质量问题，通过强制自检和一致性审核，确保交付物完整、一致、符合预期。

Unlike `/项目状态`：本 command 不只是读取和展示，还会主动发现问题并推进修复。Unlike `/进度同步`：本 command 不只是记录进度，还会审核已有进度是否真实准确。Unlike `@pm`：本 command 不管理 Task 生命周期，只专注于将当前 Task 推向完成。
</content>
</section>

<section id="scope">
<title>职责边界（In-Scope）</title>
<content>
**本 command 负责的范围**：
1. 读取并分析当前项目状态（TASK.md / progress-log.md / CLAUDE.md）
2. 自检未尽完善之处（文档缺失 / 代码不完整 / 测试未覆盖 / 边界未处理）
3. 审核一致性（CLAUDE.md vs 实际代码 / Task 状态 vs 进度记录 / 方案 vs 实现）
4. 识别当前活跃 Task 或用户指定 Task 的剩余工作
5. 按优先级排序发现的问题（must-fix / should-fix / nice-to-have）
6. 向用户汇报发现的问题并获得确认
7. 派发合适的 Agent 逐项修复
8. 修复后再次验证，循环直到达到 DoD 或用户喊停
</content>
</section>

<section id="not-scope">
<title>不承担的职责（Out-of-Scope）</title>
<content>
1. **启动全新的 Task**：只推进已有 Task，不创建新 Task → 这是 `@pm` 的职责
2. **架构级变更决策**：如发现架构问题需变更 → 升级给 `@architect`，本 command 不自行决定
3. **超出 DoD 的完美主义**：DoD 已满足后不再继续优化 → 铁律 4
4. **重写整个模块**：如发现需大规模重写 → 建议用户走 `/架构重构` 或 `/新功能`
5. **替代质量闭环**：本 command 做自检，但正式的 code-review / security-auditor / test-lead 仍按 Harness 流程执行
</content>
</section>

<section id="skill-tree">
<title>技能树</title>
<content>
**一级：状态检索与分析**
- 二级：多源状态聚合（TASK.md / progress-log.md / CLAUDE.md / Git diff）
  - 三级：Task 状态解析、进度记录交叉验证、最近变更提取、分支状态检查
- 二级：DoD 符合性检查（方案要求的验收标准是否全部满足）
  - 三级：DoD 逐项核对、未达标项标记、验收证据收集

**一级：一致性审核**
- 二级：文档-代码一致性（CLAUDE.md 描述 vs 实际代码）
  - 三级：接口契约核对、文件索引准确性、架构描述符合度
- 二级：状态-记录一致性（Task 状态 vs progress-log 记录）
  - 三级：完成标记验证、调度记录完整性、时间线合理性
- 二级：方案-实现一致性（dev-lead 方案 vs 实际代码）
  - 三级：In-scope 逐项核对、Out-scope 边界检查、接口契约符合度

**一级：缺陷识别与推进**
- 二级：未尽完善检测（文档缺失 / 代码不完整 / 测试未覆盖 / 边界未处理）
  - 三级：5-State 检查（前端）、错误路径检查（后端）、边界值检查（通用）
- 二级：优先级排序与修复编排
  - 三级：must-fix / should-fix / nice-to-have 分级、修复顺序编排、Agent 选派
</content>
</section>

<section id="methodology">
<title>方法论与执行步骤</title>
<content>
**Workflow — 6 步闭环流水线**

**Step 1 — 状态检索**（主进程执行）
- 确定 target Task：
  - 若 `$ARGUMENTS` 指定了 Task ID → 以该 Task 为目标
  - 否则 → 从 TASK.md 找出当前最活跃的 Task（状态为 in-progress 的最新 Task）
- 读取相关文档：
  - `tasks/TASK.md`：Task 描述、状态、DoD、负责 Agent
  - `progress-log.md`：最近 20 行调度记录
  - 相关 CLAUDE.md：Task 涉及的目录层的 CLAUDE.md
  - `git diff` / `git status`：未提交的变更
- 若无活跃 Task → STOP，建议用户先启动一个 Task

**Step 2 — 自检未尽完善**（主进程执行）
- 按维度逐项检查：
  1. **文档完整性**：是否缺少 API 文档 / 架构说明 / 使用手册？
  2. **代码完整性**：是否所有 In-scope 项已实现？错误路径是否处理？边界值是否覆盖？
  3. **测试覆盖**：单元测试 / 集成测试是否覆盖主要路径？是否有 flaky test？
  4. **前端 5-State**：initial / empty / loading / success / error 是否全部实现？
  5. **一致性**：CLAUDE.md 描述是否与实际代码一致？接口契约是否兑现？
  6. **技术债**：是否有临时方案（TODO / FIXME / HACK）未清理？
  7. **可维护性**：是否有重复代码 / 命名不一致 / 缺乏注释的复杂逻辑？
- 每项标记：must-fix（阻碍验收）/ should-fix（影响质量）/ nice-to-have（锦上添花）

**Step 3 — 一致性审核**（主进程执行）
- **文档-代码一致性**：读取 CLAUDE.md 中描述的接口/架构，与代码实际对比
- **方案-实现一致性**：读取 dev-lead 的 scheme，核对是否所有 In-scope 项已实现
- **状态-记录一致性**：核对 TASK.md 状态与 progress-log 记录是否矛盾
- 发现不一致 → 标记为 must-fix，记录具体差异

**Step 4 — 向用户汇报**（主进程执行，强制停顿）
- 汇总发现的问题清单：
```
## 自检报告：[Task ID]

### must-fix（阻碍验收）
- [问题 1]：{描述} | 建议派：{@agent}
- [问题 2]：{描述} | 建议派：{@agent}

### should-fix（影响质量）
- [问题 3]：{描述} | 建议派：{@agent}

### nice-to-have（锦上添花）
- [问题 4]：{描述}

### 一致性审核
- 文档-代码：{一致 / N 处不一致}
- 方案-实现：{一致 / N 处不一致}
- 状态-记录：{一致 / N 处不一致}
```
- **等待用户确认**：用户可选择：
  - "全部修复" → 进入 Step 5，按优先级顺序修复
  - "只修 must-fix" → 只修复 must-fix 项
  - "跳过某项" → 标记为跳过，不修复
  - "停止" → 中止，记录自检结果到 progress-log

**Step 5 — 逐项修复**（按优先级串行派发 Agent）
- must-fix 项按影响范围排序，逐项派发对应 Agent：
  - 代码不完整 / 边界未处理 → `@backend` / `@frontend`
  - 文档缺失 → `@doc-writer`
  - 测试缺失 → `@test-func`
  - 架构不一致 → `@dev-lead` 重新出方案
- 每项修复完成后 → 主进程快速验证 → 通过则标记完成，不通过则返回修复 Agent
- 所有 must-fix 完成后 → 询问用户是否继续 should-fix

**Step 6 — 归档与汇报**
- 将自检结果和修复记录追加到 `progress-log.md`
- 更新 TASK.md 状态（如所有 must-fix 完成 → 标记为 READY-FOR-REVIEW）
- 输出最终报告：修复了 N 项 / 剩余 M 项 / 下一步建议

**关键决策点**：
- 无活跃 Task → STOP
- 自检无问题 + DoD 已满足 → 恭喜用户，建议进入 code-review
- 发现架构级问题 → 不自行修复，升级给 `@architect`
- 用户喊停 → 立即停止，记录已完成的修复

**常见反模式与对策**：
| 反模式 | 表现 | 对策 |
|--------|------|------|
| 盲目继续 | 不检查状态直接派 Agent 干活 | 铁律 1：必须先读状态 |
| 遗漏边界 | 只检查主路径，忽略错误路径 | 维度 2 强制检查错误路径和边界值 |
| 过度优化 | DoD 已满足还继续改 | 铁律 4：DoD 满足即停 |
| 忽视不一致 | 发现文档和代码不一致但不管 | 铁律 3：一致性审核 mandatory |
| 并行修复 | 同时派多个 Agent 修不同问题 | 铁律 6：一项一项来 |
</content>
</section>

<section id="collaboration">
<title>协作边界</title>
<content>
**上游（谁触发本 command）**：
- 用户直接输入 `/继续任务 [可选 Task ID]`
- 用户输入："继续"、"还有什么没做完"、"检查一下还有哪里不够好"
- Task 实现完成后自检（替代"我觉得做完了"的主观判断）

**下游（本 command 派谁）**：
- 代码修复：`@backend` / `@frontend`
- 文档补齐：`@doc-writer`
- 测试补充：`@test-func`
- 方案重审：`@dev-lead`
- 架构问题升级：`@architect`

**横向衔接**：
- 与 `/项目状态`：本 command 先读取状态（类似 `/项目状态`），但会主动发现问题并推进修复
- 与 `/进度同步`：修复完成后调用 `/进度同步` 记录变更到 CLAUDE.md
- 与 `@pm`：本 command 推进单个 Task，不替代 `@pm` 的多 Task 调度管理
- 与质量闭环：本 command 的自检是"开发侧自检"，正式的 code-review / security-auditor 仍按 Harness 流程执行
</content>
</section>

<section id="dispatch-signals">
<title>调度触发信号</title>
<content>
**主进程应触发本 command 的场景**：
- 用户输入："/继续任务"、"继续"、"还有什么没做完"、"检查一下还有哪里不够好"
- Task 实现完成后，用户不确定是否真的做完了
- 会话中断后恢复，想确认上次的工作是否完整
- 准备提交 code-review 前的自检

**主进程不应触发本 command 的场景（防误派）**：
- 只想查看状态不修复 → `/项目状态`
- 想记录进度 → `/进度同步`
- 想做调度决策 → `@pm`
- 项目尚未开始任何 Task → 建议先启动 Task
</content>
