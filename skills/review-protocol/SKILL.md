# 审查协议

这是 quality-guardian 的工作方法论。对抗性思维是核心——**假设有问题直到证明没有**。

---

## 通用原则

1. **Critical 一票否决**：哪怕只有一条 Critical 问题，结论必须是"需修改"或"驳回"
2. **证据驱动**：每条发现必须有文件路径、行号、具体代码片段或场景
3. **责任隔离**：审查什么就只审查什么——代码审查不发表架构意见
4. **未覆盖明示**：因为工具/权限/文件缺失无法检查的项，在报告中显式标注

---

## 协议 1：需求审查（requirements-review）

### 目标
在架构阶段开始前，确保需求文档可以作为可靠的输入。

### 检查清单

#### 完整性
- [ ] 每个 Task 都有明确的"做什么"
- [ ] 每个 Task 都有可测试的验收标准（不接受"优化体验"这类模糊描述）
- [ ] 业务目标清晰说明

#### 可测性
- [ ] 验收标准描述的是可观察的行为/结果，不是实现方式
- [ ] 标准包含具体的输入、输出、条件
- [ ] 性能、安全、合规等隐性要求已显式列出

#### 边界
- [ ] 成功路径清晰
- [ ] 失败路径已考虑（输入非法、网络失败、权限不足等）
- [ ] 并发场景已考虑（如适用）
- [ ] 特殊输入已考虑（空、极大、特殊字符、多语言）

#### 依赖
- [ ] Task 间的依赖关系正确标注
- [ ] 依赖的外部系统/服务已识别
- [ ] 与已有功能的冲突已识别

#### 风险
- [ ] 技术难点已标出
- [ ] 风险有缓解建议

### Critical 示例

✗ "提升搜索体验" —— 不可测，驳回
✗ "与订单模块对接" —— 没说明如何对接，需补充
✗ "支持导入" —— 没说明导入格式、大小限制、失败处理

### 输出
`.claude/artifacts/review-requirements-{task-id}.md`

## 协议 2：架构审查（architecture-review）

### 目标
在实现阶段开始前，确保架构设计和 scope-lock 可以作为可靠的实现指导。

### 检查清单

#### 技术选型
- [ ] 使用了已有技术栈还是引入了新的？新的是否必要？
- [ ] 新依赖的版本和许可证检查
- [ ] 是否过度工程（用了复杂架构解决简单问题）
- [ ] 是否欠工程（简单方案解决了需要严谨的问题）

#### 接口契约
- [ ] 类型定义完整（没有 `any`、没有缺失的字段）
- [ ] 错误路径有明确的错误码/类型
- [ ] 边界值已声明（长度上限、范围）
- [ ] 与现有 API 风格一致

#### scope-lock 质量（关键）
- [ ] 修改范围精确到文件和函数级别
- [ ] 禁止事项充分覆盖可能的越界
- [ ] 接口契约与架构设计一致
- [ ] 实现要点具体可执行
- [ ] 验证方式包含可运行的命令
- [ ] 完成标准完整（可逐条勾选）

#### 安全
- [ ] 认证、授权在设计中已考虑
- [ ] 输入验证点已标识
- [ ] 日志脱敏已要求
- [ ] 密钥/凭证管理方式已说明

#### 可观测性
- [ ] 关键路径有日志/指标要求
- [ ] 错误情况有追踪方式

#### 并行执行图
- [ ] scope-lock 间依赖关系明确
- [ ] 可并行的任务确实独立

### Critical 示例

✗ scope-lock 说"修改 auth 模块" —— 不够精确，驳回
✗ 接口定义使用 `any` —— 类型安全失守
✗ 未禁止修改 migration 文件，但 scope 只是加字段 —— 禁止事项不充分

### 输出
`.claude/artifacts/review-architecture-{task-id}.md`

## 协议 3：代码审查（code-review）

### 目标
确保实现符合 scope-lock、安全、质量标准。

### 检查清单

#### scope 合规（最高优先级）
- [ ] 修改范围与 scope-lock 白名单**完全一致**——多一个少一个都是问题
- [ ] 没有触碰禁止事项
- [ ] 没有引入 scope-lock 未列出的依赖
- [ ] 接口契约完全匹配（函数签名、类型、返回值）

#### 安全（参考 security-checklist Skill）
- [ ] 无硬编码密钥、密码、token
- [ ] 所有外部输入经过验证
- [ ] SQL 使用参数化查询
- [ ] 权限检查在业务逻辑之前
- [ ] 日志不含敏感信息
- [ ] 错误信息不泄露内部结构

#### 错误处理
- [ ] 没有空 catch
- [ ] 错误有上下文信息
- [ ] 失败快于继续（输入非法立即拒绝）
- [ ] 资源在异常路径也能释放

#### 性能
- [ ] 无 N+1 查询（后端）
- [ ] 无不必要的重渲染（前端）
- [ ] 大对象/长列表有分页或流式
- [ ] 热路径日志级别合理

#### 测试
- [ ] 测试覆盖 scope-lock 指定的所有场景
- [ ] 边界情况有测试
- [ ] 测试本身有意义（不是只为了覆盖率）

### Critical 示例

✗ 修改了 scope-lock 禁止事项中的文件 —— 越界，驳回
✗ 引入了新的第三方包（scope-lock 未授权）—— 越界
✗ SQL 字符串拼接用户输入 —— 严重安全漏洞
✗ 密码明文存储 —— 严重安全漏洞

### 输出
`.claude/artifacts/review-code-{task-id}-{n}.md`

## 协议 4：功能测试（functional-test）

### 目标
验证实现满足需求的验收标准。

### 执行步骤

#### 1. 运行测试套件
```bash
# 从项目 CLAUDE.md 获取测试命令
{project test command}
```
记录：通过数、失败数、覆盖率

#### 2. 对照验收标准
逐条检查 requirements 中的验收标准是否满足。
使用 Grep / Read 验证代码确实实现了对应行为。

#### 3. 构造边界用例
即使单元测试都通过，也要手动设计：
- 空输入（空字符串、空数组、null、undefined）
- 极限输入（超长字符串、超大数字、负数）
- 并发场景（如适用，模拟快速连续请求）
- 权限失败（无权限用户尝试操作）
- 网络/存储失败（如有 mock 能力）

#### 4. 集成验证（如适用）
- UI 改动：用 Playwright MCP 截图并对比原始设计
- API 改动：用 curl / Postman 验证响应
- 数据改动：查询数据库验证数据状态

#### 5. 回归检查
运行全部测试，确认新改动没有破坏现有功能。

### Critical 示例

✗ 测试套件有 3 个失败 —— 驳回
✗ 验收标准"处理并发请求"未被覆盖 —— 需补充
✗ UI 截图与设计差异明显 —— 需修改

### 输出
`.claude/artifacts/review-functional-{task-id}.md`

## 审查报告通用格式

```markdown
# Review Report — {模式}: {Task 名称}

**审查时间**: {ISO timestamp}
**审查模式**: {requirements-review / architecture-review / code-review / functional-test}
**审查对象**: {文件路径列表}
**关联文档**: {其他 artifact 引用}

## 结论
{通过 ✓ / 需修改 ⚠ / 驳回 ✗}

{一句话总结}

## Critical（必须修复）
1. **[文件路径:行号]** {问题描述}
   - **证据**：
     ```{language}
     {代码片段}
     ```
   - **影响**：{如不修会发生什么}
   - **建议修复**：{具体操作}

## Warning（应该修复）
（相同格式）

## Suggestion（建议改进）
（简略格式）

## 验证通过的检查项
- ✓ {检查项}
- ...

## 未覆盖的检查项
- {因 X 原因未能检查}
```
