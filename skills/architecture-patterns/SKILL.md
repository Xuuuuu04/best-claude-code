# 架构设计模式

---

## 设计原则

### SOLID

- **S - 单一职责**：一个模块/类/函数只有一个改变的理由
- **O - 开闭原则**：对扩展开放，对修改关闭（通过抽象和插件化）
- **L - Liskov 替换**：子类可以替换父类使用而不破坏程序
- **I - 接口隔离**：多个专门接口优于一个通用接口
- **D - 依赖倒置**：依赖抽象而非具体实现

### DRY / WET / AHA

- **DRY**（Don't Repeat Yourself）：**同一业务语义的**重复必须消除
- **WET**（Write Everything Twice）：看起来相似但语义不同的重复**允许存在**（过早抽象比重复更糟）
- **AHA**（Avoid Hasty Abstractions）：抽象应该在第 3 次重复后才出现

### 错误倾向于隔离

- **错误传播不应该是默认行为**——一个模块的失败不应该无保护地让整个系统崩溃
- 每个模块边界都应该有明确的错误处理策略：fail fast、降级、重试、熔断

## 分层架构

### 经典三层

1. **Presentation**（表示层）：UI / API 端点
2. **Business Logic**（业务逻辑层）：领域逻辑、用例
3. **Data Access**（数据访问层）：DB / 第三方服务

每层只能调用**下一层**，不可反向或跨层。

### 六边形架构（Ports & Adapters）

- **核心**：领域逻辑，不依赖任何外部概念（HTTP、DB 都不出现）
- **端口**：核心需要的抽象接口
- **适配器**：外界到端口的具体实现

优势：核心可独立测试；替换基础设施不影响业务

### Clean Architecture

- Entity：业务实体
- Use Case：业务用例
- Interface Adapter：UI / 控制器 / Gateway
- Framework & Drivers：具体框架、数据库

依赖方向：始终向内（Framework 依赖 Interface Adapter，后者依赖 Use Case，最终依赖 Entity）

## 模块划分

### 按业务能力划分（推荐）

```
src/
├── order/           # 订单
├── user/            # 用户
├── payment/         # 支付
```

模块内部高内聚（所有订单相关在一起），模块间低耦合（订单不直接改用户数据）。

### 按技术层划分（不推荐作为一级划分）

```
src/
├── controllers/
├── services/
├── models/
```

问题：改一个业务要在多个目录来回跳，模块边界模糊。

**混合**：一级按业务，内部按技术层

```
src/
├── order/
│   ├── controller.ts
│   ├── service.ts
│   ├── repository.ts
│   └── types.ts
```

## 通信模式

### 同步通信

- REST API：简单、广泛支持、浏览器友好
- GraphQL：灵活查询、避免多次往返、适合复杂 UI
- gRPC：高性能、类型安全、适合内部服务

### 异步通信

- 消息队列（Kafka / RabbitMQ / SQS）：解耦、削峰、异步任务
- 事件总线：领域事件驱动的架构
- Webhook：通知第三方

### 选择依据

| 需求 | 选择 |
|:--|:--|
| 简单 CRUD、浏览器直接调用 | REST |
| 前端需要自由组合数据 | GraphQL |
| 内部服务高性能 | gRPC |
| 解耦、异步处理 | 消息队列 |
| 通知第三方 | Webhook |

## 数据一致性

### 强一致性
- 单库事务：ACID 保证
- 分布式事务：XA / Saga（慎用，复杂度高）
- 适用：金融、账户、库存核心场景

### 最终一致性
- 事件驱动
- 幂等消费 + 重试
- 适用：非关键路径、大部分业务

### CAP 权衡
- CP：一致性优先，可能降低可用性
- AP：可用性优先，最终一致
- 不同系统选择不同（支付 CP；社交 AP）

## 接口设计

### 向后兼容

- 新增字段：可选字段默认为安全值
- 修改字段：先双写双读，再切换，再下线
- 删除字段：标记 deprecated → 等待客户端升级 → 下线
- 破坏性变更：版本号升级

### 错误契约

- 错误码稳定（客户端可依赖）
- 错误码有命名空间（避免全局冲突）
- 错误响应统一结构
- 特殊情况有明确处理方式（限流、鉴权失败、资源不存在）

### 类型和约束

- 所有字段类型显式
- 枚举值显式（不要用 magic number）
- 长度/范围/格式约束在接口层声明
- Null / 缺失 / 空字符串的语义区分

## 性能考量

### 读多写少
- 缓存（Redis、CDN）
- 读副本
- 物化视图

### 写多读少
- 批处理
- 异步队列
- 分片

### 超大数据量
- 分页（游标分页 > offset 分页）
- 流式处理
- 冷热数据分离

### 高并发
- 异步处理
- 限流 / 熔断
- 无状态化（便于水平扩展）

## 可观测性设计

在设计阶段就考虑：

- **日志点**：关键业务事件、错误、慢操作
- **指标**：业务指标（下单数、成功率）+ 系统指标（延迟、错误率）
- **追踪**：跨服务的 trace ID 贯穿

**不要**等系统出问题再加可观测性。

## 常见陷阱

### 过度工程
- 为了"扩展性"添加了永远用不上的抽象层
- 为单机系统引入了分布式架构
- 为简单需求使用了复杂的消息队列 / CQRS

### 欠工程
- 核心金融操作没有事务
- 高并发场景没有幂等设计
- 缺乏可观测性，出问题无法定位

### 架构到实现的断层
- 架构图看起来合理，但 scope-lock 没有落地
- 接口契约描述模糊，开发者自行发挥

## 产出 scope-lock 的核心要求

一份好的 scope-lock 应该：

1. **精确到文件和函数**：不接受"修改 auth 模块"
2. **显式的禁止事项**：不只列白名单，也列黑名单
3. **完整的接口契约**：类型定义、错误码、边界值
4. **可执行的验证命令**：开发者可以复制粘贴运行
5. **可逐条勾选的完成标准**：不留解释空间

对照 `architecture-review-protocol` Skill 的架构审查检查清单自检。

## 决策框架

面对多个方案时：

1. **必须满足的约束**是什么？（硬约束）
2. **偏好的取舍**是什么？（软约束）
3. 各方案如何满足/违反约束？
4. 长期维护成本？
5. 可逆性？不可逆的决策要更保守

**写入 ADR（架构决策记录）**：
- 背景
- 选项
- 选择
- 理由
- 代价

下次遇到类似问题时，ADR 让你不必重新推导。
