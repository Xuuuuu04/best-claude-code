# Style Inheritance Notes

Word/DOCX 文档的样式继承是大多数交付返工的根因。本文为 docx-workflow 提供具体决策表。

## 核心原则

**改样式而不是新建样式**。客户文档已有的视觉一致性来自命名样式（"标题 1"/"正文"/"引用"），创建外观相似但名称不同的新样式会破坏目录、导航和后续维护。

## 决策表

| 场景 | 正确做法 | 错误做法 |
|:--|:--|:--|
| 用户要求"把所有标题字号改大" | 修改"标题 1"/"标题 2"样式定义 | 在每个标题段落上覆盖字号属性 |
| 用户要求"加粗第 3 章某段" | 用直接格式（local override） | 创建"加粗段落"新样式 |
| 用户要求"统一引用文献样式" | 复用或修改"引用"样式 | 新建"参考文献样式 v2" |
| 模板已有"封面副标题"但样式名为英文 | 沿用英文名 | 重命名为中文导致 TOC 失效 |
| 收到客户公司模板 | **先 dump 已有样式列表**再决策 | 直接新建样式 |

## 内容编辑 vs 格式编辑必须分开汇报

执行 docx 任务时分两类汇报，避免客户 review 时混淆"我要的内容改动"和"附带的格式变更"：

- **内容性编辑**：增删段落、改文字、调整顺序
- **格式性编辑**：样式定义变更、字号/颜色/缩进、页眉页脚、TOC 重建

## 修订与批注

- 客户开"修订模式"提交的文档必须**保留修订标记**返回，不要 accept all
- 批注是讨论而非指令，未确认前不删
- 如果客户要求"清理批注"，先确认是 resolve 还是 delete

## 验证清单（每次交付前）

- [ ] 样式列表与原模板一致（无新增冗余样式）
- [ ] TOC（目录）字段已 F9 刷新
- [ ] 修订/批注按客户要求处理
- [ ] 无残留 "样式1" / "Style2" 等系统默认占位名
- [ ] 页眉页脚、奇偶页设置无意外变化

## 失败模式

- ❌ 用 LibreOffice 改后 Word 中显示错位（避免：尽量用 python-docx 操作 XML，不依赖 GUI 渲染）
- ❌ 字体替换静默生效（嵌入字体或在交付说明里列出依赖）
- ❌ Track Changes 残留作者信息（交付前用 `Inspect Document` 清理元数据）
