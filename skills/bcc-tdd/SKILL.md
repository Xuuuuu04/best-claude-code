# /bcc-tdd — 测试驱动开发

先写测试，看它失败，再写最小代码让它通过。如果没看到测试失败，你不知道它测的是不是对的东西。

## 铁律

```
没有失败测试，就不能写生产代码
```

先写了代码再补测试？删掉代码，从测试重来。不保留"参考"，不"改改就好"。

## Red-Green-Refactor 循环

### RED — 写一个失败测试

一个测试只测一个行为。名字要说清楚测什么。优先用真实代码，mock 是最后手段。

### 验证 RED — 跑测试，确认失败

**必须做。** 确认：
- 测试是失败（不是报错）
- 失败原因是"功能还没实现"（不是拼写错误）
- 如果测试直接通过了——说明你测的是已有行为，改测试

### GREEN — 写最小代码让测试通过

只写刚好让测试通过的代码。不加功能、不重构、不"顺便优化"。

### 验证 GREEN — 跑测试，确认全绿

**必须做。** 确认：
- 新测试通过
- 旧测试也通过
- 输出干净（无 warning / error）

### REFACTOR — 清理

只在全绿之后重构。消除重复、改善命名、提取辅助函数。保持测试绿。

### 下一个循环

重复：下一个失败测试 → 下一个行为。

## Bug 修复的 TDD

发现 bug → 写一个能复现 bug 的失败测试 → 修代码 → 测试变绿。测试同时证明修复有效且防止回归。

**绝不在没有测试的情况下修 bug。**

## 常见自我合理化（红线）

| 借口 | 现实 |
|---|---|
| "太简单不需要测试" | 简单代码也会坏。测试写 30 秒。 |
| "我先写完代码再补测试" | 测试后补直接通过，证明不了什么。 |
| "手动测过了" | 手动测无法重现、无法回归、无记录。 |
| "保留代码当参考，再从测试写" | 你会"改改就好"。删掉就是删掉。 |
| "TDD 会拖慢我" | TDD 比事后 debug 快。 |
| "先探索一下" | 探索可以。探索完删掉，从 TDD 开始。 |
| "测试难写说明需求不清" | 听测试的话。难测 = 难用 = 设计有问题。 |

## 完成前检查清单

- [ ] 每个新函数/方法都有测试
- [ ] 每个测试都先看到失败
- [ ] 每个失败原因都是"功能缺失"（不是拼写错误）
- [ ] 每次都只写了最小代码让测试通过
- [ ] 所有测试通过
- [ ] 输出干净（无 error / warning）
- [ ] 测试用真实代码（mock 仅在不可避免时使用）
- [ ] 边界情况和错误路径有覆盖

不能全部打勾？说明跳过了 TDD。从头来。

## 与 Task 系统的集成

在当前 Task 的 Execution Log 中记录 TDD 过程：
```markdown
- HH:MM TDD: RED — test_empty_email_rejected (验证失败 ✓)
- HH:MM TDD: GREEN — 添加 email 校验 (全绿 ✓)
- HH:MM TDD: REFACTOR — 提取 validate_field 辅助函数
```

## 例外情况（需用户明确同意）

- 一次性原型
- 生成的代码（codegen）
- 纯配置文件修改
