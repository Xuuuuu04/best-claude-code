# /bcc-debug — 系统化调试

随机修改浪费时间且制造新 bug。快速补丁掩盖根因。

## 铁律

```
没有根因定位，就不能尝试修复
```

如果还没完成第 1 阶段，不能提出修复方案。

## 四阶段流程

### 阶段 1：根因定位（最重要）

**修之前必须做完：**

1. **认真读错误信息** — 不要跳过。栈追踪里往往有答案。记下行号、文件路径、错误码。

2. **稳定复现** — 能可靠触发吗？步骤是什么？每次都发生吗？不能复现 → 收集更多数据，不要猜。

3. **检查最近改动** — `git diff`、最近 commit、新依赖、配置变更、环境差异。

4. **多组件系统：在每个边界加诊断**
```
对每个组件边界：
  - 记录进入的数据
  - 记录离开的数据
  - 验证环境/配置传播
  - 检查每层的状态

跑一次收集证据 → 分析证据定位断裂点 → 深入那个组件
```

5. **追踪数据流** — 坏值从哪来？谁传给它的？继续往上追，直到找到源头。在源头修，不在症状处修。

### 阶段 2：模式分析

1. **找到能工作的类似代码** — 同一代码库里有什么类似的东西是正常工作的？
2. **对比参考实现** — 如果在实现某个模式，完整读参考实现，不要略读。
3. **列出所有差异** — 工作的和坏的之间有什么不同？再小也列出来。
4. **理解依赖** — 需要什么其他组件？什么配置？什么假设？

### 阶段 3：假设与验证

1. **形成单一假设** — 明确陈述："我认为 X 是根因，因为 Y"。写下来。
2. **最小化验证** — 做最小的改动来验证假设。一次只改一个变量。
3. **验证结果** — 有效 → 进入阶段 4。无效 → 形成新假设。**不要在上面叠加更多修复。**

### 阶段 4：实施修复

1. **先写失败测试**（用 `/bcc-tdd` 的 RED 阶段）
2. **实施单一修复** — 针对已定位的根因。一次一个改动。不搞"顺便改改"。
3. **验证修复** — 测试通过？其他测试没坏？问题确实解决了？

## 3 次修复失败升级规则

如果已经尝试了 3 次修复都没解决：

**停下来。这不是 bug，是架构问题。**

信号：
- 每次修复都暴露新的共享状态/耦合问题
- 修复需要"大规模重构"
- 修一个地方，另一个地方出症状

**和用户讨论，不要继续修。**

## 红线（停下来，回到阶段 1）

如果你发现自己在想：
- "先快速修一下，之后再查"
- "改改 X 试试看"
- "一次改多个地方，跑测试"
- "跳过测试，手动验证"
- "大概是 X，修一下"
- "不完全理解，但这样可能行"
- "在第 3 次尝试了，再试一次"

**全部意味着：停下来。回到阶段 1。**

## 与 Task 系统的集成

在当前 Task 的 Execution Log 中记录调试过程：
```markdown
- HH:MM DEBUG 阶段 1: 读错误 → stack trace 指向 auth/refresh.ts:42
- HH:MM DEBUG 阶段 1: 复现 → 登录 30 min 后 100% 触发
- HH:MM DEBUG 阶段 2: 对比正常的 accessToken 刷新流程，发现缺少 header
- HH:MM DEBUG 阶段 3: 假设"缺 Authorization header" → 加上后复现消失 ✓
- HH:MM DEBUG 阶段 4: TDD RED → test_refresh_includes_header → 实施修复
```

在 Decisions 段记录根因：
```markdown
- 根因：refreshToken 请求缺少 Authorization header，服务端返回 401 后客户端误判为 token 过期
```

## 常见自我合理化

| 借口 | 现实 |
|---|---|
| "问题很简单，不需要流程" | 简单问题也有根因。流程对简单 bug 很快。 |
| "紧急情况，没时间走流程" | 系统化调试比盲猜快。紧急不是借口。 |
| "先修了再查" | 第一次修复定了基调。从头做对。 |
| "Linter 过了就行" | Linter ≠ compiler ≠ runtime。 |
| "我看到问题了，修一下" | 看到症状 ≠ 理解根因。 |

## 参考技术

- **数据流追踪**：从错误点向上追溯调用栈，找到坏值的源头
- **纵深防御**：找到根因后，在多层添加校验
- **条件等待**：用条件轮询替代固定 sleep/timeout
