  <section id="format">
    <title>结果摘要格式</title>
    <content>
<pre>
---
**执行状态**: [SUCCESS | PARTIAL | BLOCKED | FAILED]
**任务文件**: [处理的 Task 文件路径]
**产出文件**: [本次创建或修改的文件路径列表，每行一个]
**状态变更建议**: [建议将 Task 状态从 X 变更为 Y]
**执行摘要**: [2-3 句话说明做了什么]
**阻塞说明**: [仅 BLOCKED 时填写：阻塞原因 + 需要哪个 Agent 介入解决]
**后续建议**: [建议调用方接下来做什么；必要时明确推荐下游 Agent]
**进度日志条目**: [一行标准格式日志，调用方负责追加到 progress-log.md]
---
</pre>
    </content>

  <section id="pm-fields">
    <title>PM Agent 专用字段：下一步调度</title>
    <content>
<p>项目管理师 Agent 的返回结果必须额外包含以下字段，因为 PM 作为子代理无法自行派生其他 Agent，必须通过返回值告诉主进程接下来做什么：</p>

<pre>
---
**下一步调度**:
- 调用 Agent: [@agent-name，如 @后端开发师、@代码审计师]
- 指令类型: [新建 | 修改 | 修复 | 审查 | 测试 | 设计 | 训练 | 研究 | 创意 | 部署 | 文档 | 调研]
- 任务文件: [Task 文件相对路径]
- 关联文件: [需要额外传给下游 Agent 的文件路径列表]
- 特殊说明: [任何补充信息，包括跳过某节点的理由]
- 是否需要用户确认: [是 | 否。如果"是"，主进程必须先向用户展示情况并获得确认后再执行]
---
</pre>

<p>主进程收到 PM 的返回后，按照「下一步调度」字段构造任务输入协议格式的指令，派生对应的 Agent。如果「是否需要用户确认」为"是"，则先向用户展示当前状态和 PM 的建议，等待用户拍板后再执行。</p>
    </content>
  </section>

  <section id="status-semantics">
    <title>状态码语义与后续处理</title>
    <content>
<h3>SUCCESS</h3>
<p>任务全部完成，产出文件已就位。调用方更新 TASK.md 主索引中该 Task 的状态，追加进度日志，根据状态机决定下一步调用哪个 Agent。</p>

<h3>PARTIAL</h3>
<p>任务部分完成。通常因为任务过大需要多次调用，或者部分工作依赖其他 Task 的产出。调用方记录已完成的部分，评估是否可以先推进其他 Task 再回来完成剩余部分。</p>

<h3>BLOCKED</h3>
<p>任务无法继续，需要其他 Agent 的产出或人工决策才能解除阻塞。
常见阻塞场景：</p>
<ul>
<li>技术方案有疑问 → 需要开发组长修订方案</li>
<li>需求描述有歧义 → 需要项目管理师澄清</li>
<li>依赖的 Task 尚未完成 → 需要先完成依赖 Task</li>
<li>发现需要架构变更 → 需要架构师介入</li>
<li>需要学术/领域深度信息 → 需要深度研究员介入</li>
<li>需要命名/文案方向 → 需要创意策划介入</li>
</ul>

<p>调用方根据阻塞说明调用对应的 Agent 解除阻塞，然后重新调用被阻塞的 Agent。</p>

<h3>FAILED</h3>
<p>任务执行失败，通常是环境问题或不可恢复的错误。调用方记录失败原因，判断是重试、换方案、还是上报人工处理。</p>
    </content>
  </section>

[2025-04-07 15:45] [代码审计师] [TASK-001] [审查未通过] - 发现3个问题：1个高危安全问题，2个中等规范问题
[2025-04-07 16:20] [后端开发师] [TASK-001] [修复完成] - 修复所有审查问题，bcrypt替换明文比较
[2025-04-07 17:00] [代码审计师] [TASK-001] [审查通过] - 第二轮审查全部通过
[2025-04-08 10:15] [机器学习工程师] [TASK-015] [训练完成] - 完成 CTR 预估模型训练，AUC 0.82，已导出 ONNX

</pre>
