## 2.2 常见 API Quirks 分类

### 认证与授权
- **Token 过期行为**: 有些 API 在 token 过期时返回 401，有些返回 403，有些返回 400
- **Scope 变更**: 升级权限后，已发放的 token 是否需要重新获取
- **OAuth 回调**: 回调 URL 是否支持通配符，是否严格匹配

### 请求/响应
- **Content-Type 严格性**: 有些 API 严格要求 `application/json`，有些接受 `text/plain`
- **字段命名风格**: 同一产品的不同 API 可能使用 camelCase 和 snake_case 混合
- **空值处理**: `null` vs `undefined` vs 省略字段，行为可能不同
- **数组序列化**: `?tags=a&tags=b` vs `?tags[]=a&tags[]=b`

### 分页
- **分页令牌过期**: 有些分页令牌在几分钟后过期
- **总数量**: 有些 API 不返回总数量，需要额外调用
- **游标方向**: 支持向前翻页但不支持向后

### Webhook
- **重试策略**: 重试间隔、最大重试次数、退避算法
- **签名验证**: 签名算法、密钥轮换机制
- **去重**: 是否保证 at-least-once 或 at-most-once
- **顺序保证**: 事件是否按发生顺序投递

### 限流
- **限流头**: `X-RateLimit-Remaining`, `Retry-After` 等头的存在和格式
- **限流粒度**: 按 API key / 按用户 / 按 IP
- **突发容量**: 是否允许短突发超出限制

## 2.3 集成风险清单

### 技术风险
- [ ] **API 稳定性**: 过去 12 个月是否有 breaking change？
- [ ] **文档质量**: Getting Started 是否能在 30 分钟内跑通？
- [ ] **SDK 成熟度**: 官方 SDK 是否覆盖项目语言？社区 SDK 是否活跃？
- [ ] **错误处理**: 错误码是否清晰？是否有详细的错误消息？
- [ ] **测试环境**: 是否有 sandbox/test 环境？
- [ ] **版本控制**: API 是否有版本号？版本策略是什么？

### 业务风险
- [ ] **服务可用性**: SLA 是多少？历史可用性数据？
- [ ] **数据主权**: 数据存储在哪个区域？是否符合合规要求？
- [ ] **供应商锁定**: 数据是否可以导出？导出格式是否标准？
- [ ] **定价变化**: 过去 24 个月是否涨价？涨价幅度？
- [ ] **功能路线图**: 需要的功能是否在路线图？预计发布时间？
- [ ] **供应商健康度**: 融资情况？客户数量？团队规模？

### 合规风险
- [ ] **许可证**: 开源组件的许可证是否兼容？
- [ ] **数据隐私**: 是否处理 PII？是否有 DPA？
- [ ] **行业合规**: 是否需要 SOC 2 / HIPAA / GDPR 等？
- [ ] **地理限制**: 服务是否在目标地区可用？

## 2.4 SDK 选型决策树

```
是否有官方 SDK？
├── 是 → SDK 是否支持项目语言？
│   ├── 是 → SDK 最近 6 个月是否有更新？
│   │   ├── 是 → SDK 的 GitHub stars > 100？
│   │   │   ├── 是 → ✅ 推荐使用官方 SDK
│   │   │   └── 否 → ⚠️ 可用，但社区较小
│   │   └── 否 → ⚠️ SDK 可能已废弃，检查维护状态
│   └── 否 → 使用 REST API 直接调用
│       ├── REST API 文档是否完整？
│       │   ├── 是 → ✅ 使用 REST API
│       │   └── 否 → ❌ 高风险，考虑其他产品
└── 否 → 是否有活跃的社区 SDK？
    ├── 是 → 社区 SDK 的维护状态？
    │   ├── 活跃（最近 3 个月有提交）→ ⚠️ 可用，需评估代码质量
    │   └── 不活跃 → ❌ 不推荐
    └── 否 → ❌ 无 SDK，需自行封装，成本高
```

## 2.5 Quota/限制分析模板

```
## Quota Analysis: [Service Name]
**Analysis Date**: [YYYY-MM-DD]

### 硬性限制（Hard Limits）
| 限制项 | 限制值 | 达到后行为 | 是否可提升 |
|--------|--------|-----------|-----------|
| 每秒请求数 (RPS) | 100 | 429 Too Many Requests | 是，需 Enterprise |
| 每日请求数 | 100,000 | 429 + 邮件告警 | 否 |
| 单请求最大 payload | 10MB | 413 Payload Too Large | 否 |
| 并发连接数 | 50 | 新连接被拒绝 | 是，需申请 |

### 软性限制（Soft Limits）
| 限制项 | 默认值 | 可调范围 | 调整方式 |
|--------|--------|----------|----------|
| 连接超时 | 30s | 10-120s | 控制台配置 |
| 重试次数 | 3 | 0-10 | SDK 参数 |
| 批处理大小 | 100 | 1-1000 | API 参数 |

### 限制监控
- **当前用量查询**: [API endpoint 或控制台位置]
- **告警设置**: [是否支持 webhook/邮件告警]
- **用量预测**: [是否提供用量趋势图]

### 限制应对策略
1. **Rate Limiting 客户端**: 实现 token bucket 或 leaky bucket
2. **缓存**: 对读请求增加缓存层
3. **批处理**: 合并小请求为大请求
4. **降级**: 达到限制时切换到备用方案
5. **预扩容**: 在预期峰值前申请提升限制
```
