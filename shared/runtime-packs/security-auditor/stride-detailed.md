## 示例：用户注册 → 登录 → 访问资源（完整 STRIDE）

```
流程: 用户注册 → 邮箱验证 → 登录 → JWT 签发 → 访问受保护资源
组件: 注册接口、邮件服务、登录接口、JWT 服务、API 网关、资源服务
信任边界:
  [用户浏览器] → HTTPS → [CDN/WAF] → [API 网关] → [认证服务] → [资源服务] → [数据库]

SPOOFING
  威胁: 攻击者使用伪造邮箱注册，冒充他人
  攻击向量: 使用 victim@example.com 注册，拦截验证邮件（如邮箱服务器配置错误）
  likelihood: 2（需要邮件系统漏洞或 DNS 劫持）
  impact: 4（账户接管）
  当前缓解措施: 邮箱验证链接含一次性 token，24小时过期
  残余风险: MEDIUM
  建议: 验证邮件添加 IP 和设备指纹记录；异常登录触发二次验证
  测试验证: 尝试用未验证邮箱直接登录；尝试重用验证链接

TAMPERING
  威胁: 修改 JWT payload 中的用户角色
  攻击向量: 拦截 JWT，修改 role 字段，重新签名（需要密钥）
  likelihood: 1（需要密钥泄露或算法混淆）
  impact: 5（完全权限提升）
  当前缓解措施: RS256 签名，密钥仅服务端持有；显式指定 algorithms=["RS256"]
  残余风险: LOW
  建议: 添加 JWT 版本号（jti），支持 token 撤销
  测试验证: 尝试 alg:none 攻击；尝试用 HS256 和公钥签名

REPUDIATION
  威胁: 用户否认执行了敏感操作（如转账、删除数据）
  攻击向量: 声称账户被盗，否认操作
  likelihood: 3（用户经常争议操作）
  impact: 3（财务或声誉损失）
  当前缓解措施: 审计日志记录 user_id + action + timestamp + IP
  残余风险: MEDIUM（日志不包含设备指纹和行为模式）
  建议: 添加设备指纹、用户行为模式（打字节奏、鼠标轨迹）、地理位置
  测试验证: 检查审计日志完整性；验证日志不可被篡改（签名或 WORM 存储）

INFORMATION DISCLOSURE
  威胁: JWT 中泄露敏感用户信息
  攻击向量: JWT 仅 Base64 编码，客户端可解码读取 payload
  likelihood: 5（JWT 必然被客户端读取）
  impact: 2（邮箱暴露，非财务信息）
  当前缓解措施: payload 仅含 user_id 和 role，不含 PII
  残余风险: LOW
  建议: 确认所有 JWT claims 都是必要的；移除任何 PII
  测试验证: Base64 解码 JWT，检查 payload 内容

DENIAL OF SERVICE
  威胁: 暴力破解登录接口
  攻击向量: 自动化工具尝试常见密码组合
  likelihood: 4（攻击工具普及）
  impact: 3（服务降级，用户锁定）
  当前缓解措施: 登录限流 5 次/分钟/IP；账户锁定 15 分钟
  残余风险: MEDIUM（分布式攻击可绕过 IP 限流）
  建议: 添加 CAPTCHA  after 3 次失败；实施渐进式延迟；监控异常登录模式
  测试验证: 尝试 100 次/分钟登录；验证限流和锁定机制

ELEVATION OF PRIVILEGE
  威胁: 普通用户通过修改请求获得管理员权限
  攻击向量: 注册时设置 role="admin"；修改 JWT role claim
  likelihood: 2（需要应用层漏洞）
  impact: 5（完全系统控制）
  当前缓解措施: role 从数据库读取，不从请求/JWT 获取
  残余风险: LOW
  建议: 管理员角色分配需二次确认（邮件确认 + 管理员审批）
  测试验证: 尝试注册请求含 role=admin；尝试修改 JWT role claim
```

## 攻击树模板

对于 HIGH 风险发现，绘制攻击树展示发现如何启用进一步利用链：

```
[目标: 提取所有用户 PII]
  ├── [A] SQL 注入 on /api/search
  │     └── [A1] UNION SELECT 提取 users 表
  │           └── [A1a] 无输入验证 → 直接注入
  │           └── [A1b] 有 WAF → 尝试绕过编码
  ├── [B] 管理员账户接管
  │     ├── [B1] 暴力破解（无 rate limit）
  │     │     └── [B1a] 弱密码策略 → 字典攻击
  │     └── [B2] 会话固定
  │           └── [B2a] 登录后 session ID 不变
  └── [C] IDOR on /api/users/{id}
        └── [C1] 无权限检查 → 遍历所有 ID
              └── [C1a] 顺序 ID → 容易枚举
              └── [C1b] UUID → 需要信息泄露获取 UUID

优先级分析：
- 修复 [C1] 可阻断 33% 的攻击路径
- 修复 [A1] 可阻断 33% 的攻击路径
- 修复 [B2] 可阻断 17% 的攻击路径
- 建议优先修复 [C1] + [A1]（合计阻断 66%）
```

## 信任边界图模板

```
[用户浏览器]
    ↓ HTTPS (TLS 1.3)
[CDN / WAF]              ← 信任边界 1：外部 → 边缘
    ↓ HTTPS
[负载均衡器]              ← 信任边界 2：边缘 → 内部
    ↓ HTTP (内部网络)
[API 网关]               ← 信任边界 3：网关 → 服务
    ↓ gRPC (mTLS)
[微服务集群]
    ├── [认证服务]
    ├── [用户服务]
    ├── [订单服务]
    └── [支付服务]
    ↓ TCP (无 TLS)       ← 潜在发现：明文内部通信
[数据库]
    ├── [PostgreSQL 主库]
    └── [Redis 缓存]
    ↓ 备份通道
[对象存储]
    └── [S3 / MinIO]

每个信任边界 crossing 都是潜在攻击面：
- 边界 1：DDoS、WAF 绕过、TLS 降级
- 边界 2：内部网络嗅探（如果边缘被突破）
- 边界 3：服务间认证绕过
- 数据库层：SQL 注入、权限提升
```

## STRIDE 评分到 CVSS 映射参考

| STRIDE 发现 | 典型 CVSS 范围 | 说明 |
|---|---|---|
| 身份伪装（JWT 绕过） | 8.0–9.8 | 取决于获取的权限级别 |
| 数据篡改（未签名更新） | 7.0–9.0 | 取决于可篡改的数据 |
| 抵赖（无审计日志） | 4.0–6.0 | 通常是 Medium |
| 信息泄露（IDOR） | 5.0–8.0 | 取决于数据敏感度 |
| DoS（无限流） | 4.0–7.5 | 取决于影响范围 |
| 权限提升 | 8.0–10.0 | 几乎总是 High/Critical |
| 供应链污染 | 7.0–9.8 | 取决于污染范围 |
