# Domain 1: 需求语义增强与 Brief 模板

## 1.1 语义增强技术

### 模糊表达识别词典

| 模糊表达 | 潜在含义 | 澄清问题 |
|---------|----------|----------|
| "简单做一下" | 范围未定 / 快速交付 / 易用性 | "核心功能是哪 3-5 个？谁是主要用户？" |
| "有点像 xxx" | 参考产品 | "xxx 的哪些功能需要？哪些不需要？有什么不同？" |
| "加个 AI 功能" | 能力类型不明 | "具体是聊天机器人、推荐引擎、图像识别还是其他？" |
| "做个 APP" | 平台未定 | "iOS？Android？小程序？H5？都需要？" |
| "能不能快点" | 时间压力 | "硬截止日期是什么？哪些可以延后？" |
| "差不多就行" | 质量标准不明 | "哪些是必须精确的？哪些可以简化？" |
| "最好有 xxx" | 优先级不明 | "'最好有' = Must Have 还是 Could Have？" |
| "我们可能需要" | 范围膨胀信号 | "'可能' 的概率是多少？不做的影响是什么？" |

### 竞争参考分解法

当客户说 "像 Notion/LinkedIn/Slack" 时：

```
步骤 1: 列出参考产品的核心功能
Notion:
- 文档编辑 (块编辑器)
- 数据库 (表格/看板/日历视图)
- 团队协作 (评论/提及)
- 模板系统
- 权限管理
- 集成 (API/Webhook)

步骤 2: 逐一确认
"您提到的 Notion，以下功能哪些需要包含？"
| 功能 | 需要？ | 优先级 | 备注 |
|------|--------|--------|------|
| 文档编辑 | Yes/No | Must/Should | |
| 数据库 | Yes/No | Must/Should | 哪种视图？ |
| 团队协作 | Yes/No | Must/Should | |
| ... | | | |

步骤 3: 确认差异化
"与 Notion 相比，您的产品有什么不同？"
- 目标用户不同？
- 核心场景不同？
- 必须避免的感觉？
```

### 隐含需求挖掘清单

以下需求客户通常不会主动提及，但必须确认：

- [ ] **认证与授权**: 登录方式？权限粒度？
- [ ] **移动端适配**: 是否需要响应式？原生 App？
- [ ] **数据导出**: 用户能否导出数据？格式？
- [ ] **管理后台**: 是否需要管理员界面？
- [ ] **日志与监控**: 是否需要操作日志？错误监控？
- [ ] **备份与恢复**: 数据备份策略？RTO/RPO？
- [ ] **多租户**: 是否需要支持多个组织/团队？
- [ ] **国际化**: 是否需要多语言？多时区？
- [ ] **合规**: 是否需要 GDPR/等保/其他合规？

## 1.2 Brief 结构化模板

### 标准 Client-Brief 结构

```
# Client Brief: [Project Name] v[N]
**Date**: [YYYY-MM-DD]
**Analyst**: [@client]
**Status**: Draft / Ready for @pm

## 1. 项目背景
- 客户业务: [一句话描述]
- 目标用户: [具体描述]
- 核心痛点: [客户试图解决什么问题]

## 2. 核心功能 (语义增强后)
### 2.1 [功能模块 1]
- 来源: [CLIENT STATED / INFERRED]
- 用户故事: As a [role], I want [feature] so that [value]
- 验收标准: Given [context], When [action], Then [outcome]
- 优先级: Must / Should / Could / Won't
- 状态: Confirmed / Pending Clarification

### 2.2 [功能模块 2]
...

## 3. 用户角色
| 角色 | 场景 | 频率 | 重要性 |
|------|------|------|--------|
| [角色 1] | [场景] | [高频/中频/低频] | [高/中/低] |

## 4. 非功能需求
- 性能: [具体指标，如 "页面加载 < 2s"]
- 安全: [认证方式，数据加密要求]
- 合规: [GDPR/等保/其他]
- 可用性: [SLA 要求]

## 5. 时间期望
- 客户期望: [原话]
- 评估范围: [X-Y 周/月]
- 关键里程碑: [如有]

## 6. 预算
- 客户预算: [原话]
- 范围匹配度: [匹配/紧张/不匹配]
- 建议: [如需要调整范围或预算]

## 7. Out-of-Scope 锚点
- [明确排除的功能 1]
- [明确排除的功能 2]
- [明确排除的功能 3]

## 8. 待澄清事项
1. [问题] — 阻塞: [什么决策]
2. [问题] — 阻塞: [什么决策]

## 9. 技术可行性
- 评估: [Conventional / Needs pre-research / Fundamentally infeasible]
- 需预研项: [如有]

## 10. 风险登记
| # | 类型 | 描述 | 概率 | 影响 | 缓解 |
|---|------|------|------|------|------|
| 1 | | | | | |

## 11. Go/No-Go 评估
- 结论: [GO / CONDITIONAL GO / NO-GO]
- 理由: [2-3 句]
- 下一步: [@pm / @tech-research / 客户澄清]
```

## 1.3 行业术语翻译表

### 通用企业软件

| 客户术语 | 可能含义 | 澄清问题 |
|---------|----------|----------|
| "CRM" | 联系人管理 + 销售漏斗 + 活动记录 + 报表 | "需要哪些子模块？" |
| "ERP" | 采购 + 库存 + 财务 + HR | "核心模块是哪个？" |
| "OA" | 审批流 + 日历 + 文档管理 | "审批复杂度如何？" |
| "中台" | 数据中台 / 业务中台 / 技术中台 | "具体是哪类中台？" |
| "大数据" | 报表看板 / 数据仓库 / 实时分析 | "数据量和实时性要求？" |
| "云服务" | 云托管 / 云原生架构 / SaaS | "具体需求是什么？" |
