# Domain 2: 售前评估框架与提案模板

## 2.1 Go/No-Go 评估矩阵

### 标准评估矩阵

```
## Go/No-Go Evaluation Matrix

| 维度 | 权重 | 评分 (1-5) | 加权分 | 评估标准 |
|------|------|-----------|--------|----------|
| 技术可行性 | 25% | | | 5=常规开发; 1=技术不可行 |
| 商业可行性 | 25% | | | 5=预算充裕; 1=预算严重不足 |
| 执行匹配度 | 25% | | | 5=团队经验丰富; 1=无相关经验 |
| 客户关系 | 25% | | | 5=决策清晰，沟通顺畅; 1=决策混乱 |
| **总分** | 100% | | | |

评分标准:
- 4.0-5.0: GO
- 3.0-3.9: CONDITIONAL GO (pending specific conditions)
- 1.0-2.9: NO-GO
```

### 各维度评估检查清单

**技术可行性 (Technical Feasibility)**
- [ ] 核心功能是否在当前技术能力范围内？
- [ ] 是否有新技术风险？（AI/ML、区块链、实时处理等）
- [ ] 性能要求是否现实？
- [ ] 集成复杂度是否可控？
- [ ] 是否需要 @tech-research 预研？

**商业可行性 (Commercial Viability)**
- [ ] 预算是否与范围匹配？
- [ ] 付款条款是否合理？
- [ ] 是否有隐性成本？（第三方服务、合规认证等）
- [ ] 利润率是否可接受？
- [ ] 是否有长期价值？（维护合同、扩展机会）

**执行匹配度 (Execution Fit)**
- [ ] 团队是否有类似项目经验？
- [ ] 当前资源是否充足？
- [ ] 时间线是否现实？
- [ ] 技术栈是否匹配？

**客户关系 (Relationship Quality)**
- [ ] 是否有明确的决策人？
- [ ] 决策流程是否清晰？
- [ ] 客户是否 responsive？
- [ ] 是否有历史合作经验？
- [ ] 客户期望是否现实？

## 2.4 沟通话术指南

### 坏消息传递框架

```
1. 承认 (Acknowledge)
   "我们理解这会对您的计划产生影响..."

2. 解释原因 (Explain — no blame)
   "由于 [客观原因，如第三方 API 变更]，我们需要调整..."
   避免: "因为你们需求变了" / "因为开发太慢"

3. 说明行动 (State action)
   "我们正在采取以下措施: (1)... (2)..."

4. 给出新时间线 (New timeline — range)
   "预计调整后的完成时间是 [X-Y 天/周]"

5. 提供决策点 (Decision point)
   "您希望 (A) 等待完整修复 还是 (B) 先上线部分功能？"
```

### 技术-业务翻译示例

| 技术表达 | 业务翻译 |
|---------|----------|
| "第三方 API 废弃了我们依赖的 webhook endpoint" | "支付通知系统需要更新 — 预计需要 3 天，建议在下个账单周期前完成" |
| "数据库查询存在 N+1 问题" | "搜索功能在数据量大时会变慢，我们需要优化 — 预计 2 天" |
| "需要实现分布式锁" | "为了防止同时编辑导致的数据冲突，我们需要增加一个保护机制 — 预计 1 天" |
| "JWT token 需要刷新机制" | "为了安全，登录状态需要定期刷新 — 用户可能需要重新登录一次" |

### 期望管理话术

**当客户期望过高时**:
```
"我理解您希望 [期望]。基于我们的经验，[现实情况]。
为了达到最好的结果，我建议 [替代方案]。
这样我们可以在 [时间] 内交付 [价值]，同时为 [未来扩展] 打下基础。"
```

**当客户要求加急时**:
```
"我们可以评估加急方案。通常加急意味着：
(1) 增加资源投入，成本增加 [X]%
(2) 或缩减范围，先交付核心功能
(3) 或接受一定质量风险
您倾向于哪种方式？"
```

**当客户提出超出范围的需求时**:
```
"这是一个很好的想法。根据我们的合同，这属于 [新功能/扩展功能]。
我可以为您评估：
(1) 添加到当前项目的影响（时间和成本）
(2) 作为独立 Phase 2 项目的影响
您希望我先做哪个评估？"
```
